# Group g03 — Ephemeral verification — [[1.1.2-establish-prisma-database]]

- **Order:** 3.
- **Depends on:** g02.
- **Description:** Prove migration replay, constraints, runtime CRUD, and effective least privilege on a disposable PostgreSQL service.

## Movements

- [x] (agent) Add a Node test orchestrator for `pnpm db:test`: start a uniquely named `postgres:18.4-bookworm` container with trust auth only on a random loopback host port, wait for SQL readiness, and register unconditional `finally`/signal cleanup with volume removal.
  - Expected: concurrent runs do not collide; the admin URL is process-local; success, assertion failure, and interruption all remove the container; no fixed/test credential is committed.
  - If it fails: orphan cleanup or port collision is possible → correct naming/port discovery and add a failure-path self-test before proceeding; never use a developer's existing database.
- [x] (agent) Have the orchestrator apply bootstrap as container admin; connect as `qr_migrator`, assert `current_user`, effective database `CONNECT`, schema ownership, and absence of superuser/role-admin attributes; then run `prisma migrate deploy` with only `MIGRATION_DATABASE_URL`, verify migration status/history/checksum state, and deploy again.
  - Expected: the real migrator connection prints `PASS migrator-connect`; fresh deploy through that URL applies exactly one committed migration and prints `PASS migration-replay`; second deploy reports no pending migration and prints `PASS migration-idempotence`.
  - If it fails: role cannot own/alter an expected object → inspect committed SQL and exact ownership, then use only an authorized fork; history/drift ambiguity → abort.
- [x] (agent) Through `DATABASE_URL` and the generated Prisma client, insert/select/update/delete a fixture row using the identity default; inspect catalog schema, columns, types, defaults, named PK/unique/check constraints, and owner.
  - Expected: DML succeeds and rolls back/cleans up; schema matches exactly; print `PASS foundation-schema` and `PASS runtime-crud`.
  - If it fails: missing table/sequence grant → correct current/default grants narrowly; schema mismatch → repair schema and the not-yet-released baseline migration together.
- [x] (agent) Execute negative inserts and assert SQLSTATE plus constraint name for null key (`23502`), duplicate key (`23505`), blank key and negative quantity (`23514`); reject any other error.
  - Expected: every invariant is database-enforced and prints `PASS constraint-sqlstates`; no probe row remains.
  - If it fails: localized message matching or broad catch masks the error → use driver error `code`, `constraint`, and `column`; constraint does not fire → abort and repair migration.
- [x] (agent) Query `has_*_privilege`, `pg_has_role`, owners, table/sequence ACLs, role attributes, and `_prisma_migrations` denial; then attempt runtime `CREATE TABLE app`, `ALTER TABLE`, `CREATE TEMP TABLE`, and `CREATE ROLE`, accepting only `42501` and cleaning any unexpected artifact transactionally.
  - Expected: runtime has only intended fixture DML/sequence access, cannot read or mutate migration metadata, owns nothing, cannot `MEMBER`/`SET` migrator, and every forbidden operation fails; print `PASS role-separation` and `PASS runtime-denials`.
  - If it fails: any forbidden operation succeeds → abort immediately, preserve catalog evidence, and return to the grant/bootstrap movement; never paper over it with a test exception.
