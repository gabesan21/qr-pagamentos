## Route

Set `output: "standalone"`. Add `.dockerignore` excluding Git/worktrees, `.env*` except examples, secrets, dependencies, build/test output, docs/kanban, and local noise. One multi-stage `Dockerfile` uses the verified Node tag+index digest for dependency, build, `db-ops`, and app targets. The build receives no secrets. `app` contains only standalone output plus `public`/`.next/static`, runs `node server.js` as fixed UID/GID 1000, has a Node-based liveness `HEALTHCHECK`, read-only root FS, init, and a UID-owned cache tmpfs. `db-ops` contains only frozen production/tool dependencies, Prisma config/schema/migrations/bootstrap, and container wrappers, and also runs as 1000:1000.

Add fail-closed Node wrappers: bootstrap reads three staged secret files, validates nonempty/distinct values, connects as cluster admin without logging a URL, executes existing `prisma/bootstrap.sql`, then assigns passwords using tested SQL identifier/value escaping; migrate/runtime build URLs with WHATWG setters. Migrate spawns existing `prisma migrate deploy`; runtime first opens the least-privilege runtime connection, executes `SELECT 1`, closes it, and only then spawns standalone server without a shell. A failed preflight exits before bind; after startup `/api/health` remains DB-independent. Secrets never enter Compose environment, build args, layers, command lines, generated config, or logs.

Compose defines `db`, `bootstrap`, `migrate`, and `app`; named `postgres-data`; internal `database` network for all four and an `edge` network only for app egress; no database/ops ports. Only app publishes `127.0.0.1:${APP_PORT:-3000}:3000`. Original secret sources are mode 0600, owned/readable only by the invoking Docker user. DB alone receives the original admin file through `POSTGRES_PASSWORD_FILE`: the unmodified official entrypoint starts with sufficient initialization identity, consumes it during initial cluster setup, then the PostgreSQL server must run non-root; no UID1000/remap claim applies to DB. `pnpm container:prepare-secrets` uses a pinned, no-network helper under the same daemon/user namespace to copy only required sources into per-secret staged mounts owned/readable as 1000:1000/0400; bootstrap, migrate, and app mount only their required staged files and run UID1000. Every source/staged/readability boundary is probed without content output. Bootstrap waits for DB health; migrate waits for bootstrap exit 0; app waits for migrate exit 0. Jobs use `restart: "no"`; DB/app use `unless-stopped`; app health is `/api/health`.

Add `.env.compose.example` with only nonsecret port and original secret-file paths, plus `pnpm container:prepare-secrets`, `container:contract-check`, and `container:test`. The harness creates invoking-user-owned mode-0600 reserved-character sources, stages UID1000 copies, uses a unique project, checks initial admin-secret consumption, post-start server UID, jobs/runtime preflight/app/static assets, and forces real bad-admin, bad-migrator, and bad-runtime authentication. It scans all outputs/metadata, reruns cleanly, and always removes staged copies plus `down --volumes --remove-orphans`. README documents exact production startup/inspection/stop, non-destructive rollback, cleanup, and 005-only critical verification.

## Forks

- If exact tag inspection returns a changed index digest before implementation commit → use the newly inspected digest, record tag/digest/platform evidence in execution notes, and rerun all builds; do not float.
- If `docker compose up --wait` does not propagate a failed completed job → the test harness must inspect job exit codes and app absence explicitly; keep production dependency conditions unchanged.
- If standalone tracing omits a db-ops dependency → copy/install it only in `db-ops`; never expand the app image to full `node_modules`.
- If root filesystem writes occur only under `.next/cache` or `/tmp` → add the narrow UID-owned tmpfs; any other write requires return to 002, not disabling read-only mode.

## Abort conditions

- Dependencies 1.1.1/1.1.2 or commit `8ab3f72` are absent from the worktree base.
- Any solution requires a shared/admin runtime credential, usable committed secret, secret build input, root app/db-ops process, unpinned base, public DB/ops port, or weakened 1.1.2 SQL/probes.
- Bootstrap/migration failure can still start app, application liveness is made database-aware, or retries can conceal a failed one-shot job.
- Clean-clone tests need a provider, reverse proxy, TLS, Nautt integration, or production deployment assumption.
- Existing-volume migration is destructive/non-idempotent, or rollback would silently run `down --volumes` against operator data.
