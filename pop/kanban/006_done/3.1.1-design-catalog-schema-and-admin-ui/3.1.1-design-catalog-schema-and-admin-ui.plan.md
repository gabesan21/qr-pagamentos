# 3.1.1-design-catalog-schema-and-admin-ui — Plan

## Objective

Add Prisma models and a bilingual `/admin` UI for managing Nautt currency-pair and payment-method UUID catalogs, including server-side UUID validation, active/inactive flags, and opaque protected mutation outcomes.

## Affected areas/files

- `prisma/schema.prisma` — new catalog models.
- `prisma/migrations/` — one reviewed versioned migration (do not use `db push`).
- `src/generated/prisma/` — regenerated client (`pnpm db:generate`).
- `src/auth/nautt-catalog.ts` — server-only admin-authorized list/create/update/toggle service.
- `src/app/admin/catalog/` — route handlers for catalog mutations (follow `src/app/admin/payment-settings/route.ts` guard pattern).
- `src/app/admin/page.tsx` and `src/app/admin/admin-surface.tsx` — add currency-pair and payment-method sections.
- `src/i18n/dictionaries/en.ts` and `src/i18n/dictionaries/pt-BR.ts` — new catalog labels, validation, empty/error states.
- `src/i18n/dictionaries.ts` and `src/i18n/dictionaries.test.ts` — parity check.
- `DESIGN.md` — update if the catalog UI introduces new primitives or states.
- `AGENTS.md` tree — no new child contracts required; follow `prisma/AGENTS.md` and `src/components/ui/AGENTS.md`.

## Strategy and high-level approach

1. **Schema first**: define `CatalogCurrencyPair` (`currencyUuid`, `exchangeCurrencyUuid`, `label`, `active`) and `CatalogPaymentMethod` (`paymentMethodUuid`, `label`, `active`) with UUID-typed UUID columns, unique constraints, and audit timestamps.
2. **Migration**: create a single forward migration under `prisma/migrations/`, review SQL grants/constraints, run `pnpm db:generate` and `pnpm db:test` against the disposable container.
3. **Service**: implement `src/auth/nautt-catalog.ts` with `list`, `create`, `updateLabel`, and `setActive` operations; re-authorize the admin actor on every call and return safe DTOs.
4. **Routes**: add POST handlers for create and update/toggle, reusing `requireAdminFromCookie` and `protectedMutationResponse` from `src/app/admin/guard.ts`; redirect with `?success=` / `?error=` query params.
5. **UI**: extend `AdminSurface` with two new card sections listing records and inline forms; reuse `Card`, `Field`, `Input`, `Checkbox`, `Button`, `Badge`, `Alert` from `src/components/ui/`; support empty, loading, error, and inline-confirmation states.
6. **i18n**: add dictionary keys for headings, labels, validation, empty/error/success messages, keeping `en.ts`/`pt-BR.ts` in parity.
7. **Tests**: service tests, route tests, page tests, and dictionary parity; run `pnpm lint`, `pnpm typecheck`, `pnpm test`, and `pnpm db:test`.

## Fronts/ownership

- **Schema & migrations**: prisma subtree (single owner).
- **Service + routes**: `src/auth/` and `src/app/admin/catalog/` (single owner; no separate API surface).
- **Admin UI + i18n**: `src/app/admin/` + `src/i18n/` (single owner).

All fronts are sequential: schema → migration → service → routes/UI → i18n/tests.

## Dependencies and risks

- **Dependencies**: none (task card lists `depends_on: []`).
- **Risks**:
  - Confusing the Nautt UUID catalog with the existing `GlobalPaymentSettings` BRL/PIX allowlist — keep them separate.
  - Invalid or non-canonical UUID strings reaching the database — enforce server-side UUID format validation before persistence.
  - Inactive records being selected for future products/links — the schema supports `active`; enforcement belongs to later product/link tasks, but this task must make the flag queryable and documented.
  - Existing design-system primitives may not cover table-based lists with many rows; verify inventory in `DESIGN.md` before adding new components.

## Acceptance criteria

- [ ] `prisma/schema.prisma` declares `CatalogCurrencyPair` and `CatalogPaymentMethod` with UUID columns, labels, active flags, and audit timestamps.
- [ ] A reviewed migration exists and `pnpm db:test` passes.
- [ ] `src/auth/nautt-catalog.ts` provides admin-authorized list, create, update, and toggle operations with safe DTOs.
- [ ] Admin catalog route handlers re-authorize the cookie principal and return only empty `401`/`403` protected outcomes.
- [ ] `/admin` displays bilingual create/list UI for both catalogs, reusing the existing shadcn inventory and supporting empty/loading/error states.
- [ ] All user-visible strings are added to `en.ts` and `pt-BR.ts` and pass dictionary parity tests.
- [ ] `pnpm lint`, `pnpm typecheck`, and `pnpm test` pass; `pnpm db:test` passes if schema/migration changed.
- [ ] `DESIGN.md` is updated only if a new primitive or state is introduced; otherwise the existing PIX-ledger contract remains authoritative.
