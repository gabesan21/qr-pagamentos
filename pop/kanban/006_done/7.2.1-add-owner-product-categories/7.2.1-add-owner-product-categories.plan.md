# Plan — [[7.2.1-add-owner-product-categories]]

- **Stage:** 002_planning · **Responsible:** planning agent

## Objective and expected result

- **Objective:** establish owner-scoped product categories and an opaque, concurrency-safe management boundary that can reassign current products without weakening owner isolation or historical commerce records.
- **Observable result:** an active `USER` can list and create only owned categories, edit one at its expected version, and deactivate it atomically; a referenced category requires an explicit active same-owner replacement, while missing, stale, cross-owner, inactive, and invalid replacement cases reveal no target state. No category reaches a public response or UI in this task.

## Strategy and fixed contract

Add a stable UUID category owned by one `User`, with required `pt-BR` and English names, active state, nonnegative CAS version, and timestamps. Both names follow the existing public-title boundary: Unicode-trimmed, nonblank, single-line, at most 160 code points, and persisted without case folding or internal-whitespace rewriting. Each persisted localized name is exact-unique within its owner; a collision is one opaque conflict.

Existing products receive only a nullable, same-owner category reference: all V1 rows remain uncategorized without backfill, and ordinary product create/edit/category selection remains owned by 7.2.2. Category edit and one-way deactivation use expected-version CAS. Deactivation without a replacement is allowed only when no product references the source; otherwise one explicit distinct, active, same-owner replacement is required and every source product is reassigned in the same transaction before the source becomes inactive. Source/replacement checks and writes serialize deterministically, recheck state at commit, and either all apply or none apply.

Unknown, stale, inactive, malformed, cross-owner, and deleted-owner mutations share the existing protected/opaque outcome families and perform no partial write. Category ownership and product assignment are enforced by composite owner keys and restrictive deletion. Disabled/deleted principals remain unauthenticated; hard owner deletion cannot orphan categories or product references. Inactive category rows are retained, never physically deleted, and reassignment changes only the current product association: product values, payment links, order snapshots, identifiers, provider data, and V1 read behavior remain unchanged.

## Risks and abort conditions

- **Risk:** reassignment races produce cross-owner or inactive-category references — mitigate with composite database ownership, deterministic transactional serialization, commit-time eligibility checks, and CAS/concurrency tests.
- **Risk:** a category migration rewrites V1 products or leaks into current public reads — mitigate with a nullable no-backfill reference, projection regression tests, and byte-preservation checks for existing application rows.
- **Abort if:** owner deletion, source/replacement locking, or concurrent deactivation cannot be proven all-or-nothing with the current transaction boundary; return to 002 instead of weakening opacity or history compatibility.
