## Route

Pin `prisma`, `@prisma/client`, and `@prisma/adapter-pg` at 7.8.0 plus `pg` 8.22.0 and its types. Configure PostgreSQL in `prisma/schema.prisma`, migration CLI access through `MIGRATION_DATABASE_URL` in `prisma.config.ts`, generated code in ignored `src/generated/prisma/`, and runtime access through `DATABASE_URL` in one lazy `src/db/client.ts` singleton. No URL or password is committed.

Create only `DatabaseFoundationFixture` mapped to `_database_foundation_fixture`: `id BigInt`/identity primary key, unique nonblank `key varchar(64)`, nonnegative `quantity integer`, and database-defaulted `created_at timestamptz(3)`. Commit one reviewed migration `20260714000000_foundation_baseline/migration.sql`; custom SQL names the unique/check constraints. This fixture has no route, identity, catalog, payment, seed, or business semantics and may be removed when the first real domain migration replaces its proof duty.

Provision `qr_migrator` and `qr_runtime` as separate `LOGIN`, `NOINHERIT`, `NOSUPERUSER`, `NOCREATEDB`, `NOCREATEROLE`, `NOREPLICATION`, `NOBYPASSRLS` roles with no committed passwords. After revoking public `CONNECT`, grant it explicitly to both roles. `qr_migrator` owns schema `app`; `qr_runtime` gets schema `USAGE`, application-table `SELECT/INSERT/UPDATE/DELETE`, and sequence `USAGE`, never ownership/DDL/role administration or access to `_prisma_migrations`. Revoke public `TEMPORARY` and public-schema `CREATE`; use migrator-scoped default privileges for future objects and explicitly revoke migration-metadata access. Production operators assign secrets externally.

Add a Node-controlled `pnpm db:test` harness that always starts and finally removes a fresh `postgres:18.4-bookworm` container on a random loopback port, bootstraps roles as the container admin, applies migrations using only the migrator URL, runs a second no-op deploy/status check, then runs SQL/Prisma probes using only the runtime URL. It prints one stable `PASS <contract>` line per contract. Keep `pnpm check` deterministic and database-free; CI runs `pnpm check` and `pnpm db:test` as separate required jobs/steps.

Do not change `/api/health`: it remains application liveness and makes no database/readiness claim. Document that task 1.1.3 must gate startup on PostgreSQL health and successful `migrate deploy`, not reinterpret this endpoint.

## Forks

- If PostgreSQL 18.4 cannot run on the executor architecture but official `postgres:18.4` supports it → use the matching 18.4 Debian variant and record the resolved image; do not change the major.
- If `migrate deploy` creates an object type beyond tables/sequences → extend both current and migrator-scoped default grants only for the demonstrated runtime operation, then add a denial/allowance probe in the same migration test.
- If Prisma emits migration SQL different from the reviewed fixture DDL → keep the generated structure, edit via `migrate dev --create-only`, and reconcile exact constraint names before first application; never edit an applied migration.
- If generated TypeScript is linted despite ignore rules → exclude only `src/generated/prisma/**` in ESLint; never weaken linting for handwritten `src/db/**`.

## Abort conditions

- Dependency 1.1.1 is not present at the worktree base, or any task dependency is incomplete.
- Docker/PostgreSQL cannot run after the documented image-variant fork, or a clean ephemeral database cannot be guaranteed.
- A package requires changing Node 24.18.0, pnpm 11.13.0, TypeScript 5.9.3, Next.js, or React.
- Role isolation would require superuser/`CREATEROLE`/`CREATEDB` at runtime, shared connection credentials, committed secrets, or an unreviewed provider exception.
- A required change introduces an auth, administrator, catalog, order, payment, seed, or production-container model/behavior; return to 002 to split/re-scope.
- Migration/schema drift remains after a fresh replay, or a runtime DDL/role-administration probe unexpectedly succeeds.

## Red-team

- **Attack that failed against the plan:** “Two environment variable names prove least privilege.” The plan resists by querying effective ACLs/ownership/membership and executing CREATE, ALTER, TEMP, and CREATE ROLE denials through the runtime connection.
- **Attack that failed against the plan:** “A passing second deploy hides an edited applied migration.” Every run starts a fresh container and replays only committed SQL before the no-op deploy, so missing/invalid history fails clean replay.
- **Attack that passed → patch:** Prisma-created future tables could omit runtime grants because default privileges are role-specific → bootstrap now sets defaults `FOR ROLE qr_migrator`, tests actual fixture CRUD/sequence access, and requires expansion only for demonstrated object types.
- **Attack that passed → patch:** Adding database readiness to `/api/health` would collapse three health boundaries and make builds depend on PostgreSQL → health files are explicitly untouched and criterion 7 compares them with `origin/develop`; 1.1.3 owns readiness/startup gating.
- **Round 1 critic patches:** revoking public database access also disconnected the non-owner migrator, so bootstrap now grants and probes migrator `CONNECT`; excluding `.env.example` let a usable extra credential evade the scan, so criterion 6 scans it and compares its entire content to two non-usable placeholders.
