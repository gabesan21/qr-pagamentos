## Round 1 — 2026-07-20

### Decision brief

- **Delivery:** Persist exact-decimal bilingual products and provide protected administrator CRUD controls.
- **Strategy:** Add the forward-only persistence contract, then the authorized server boundary, then the localized admin surface while preserving existing security and design-system contracts.
- **Topology:** Three sequential fronts: persistence → service/routes → UI.
- **Primary risks:** Decimal coercion, lost updates, authorization disclosure, and UI/component drift.
- **Primary criteria:** AC1–AC5 cover database constraints, protected mutations, conflict behavior, localized UI evidence, and the aggregate project gate.
- **Plan:** [[3.2.1-product-schema-and-admin-crud.plan]] — *follow for the complete execution brief*.

### Resposta do revisor (yolo)

Devolvida pelo revisor independente (yolo) — 2026-07-20.

- **Contrato decimal incompleto:** o plano exige uma string decimal positiva
  canônica, preservação exata, validação no banco e formatação localizada, mas a
  spec não define a gramática canônica nem limites de precisão/escala. Sem isso,
  F01/F02 teriam de inventar um contrato durável e os testes de AC1/AC2 não têm
  um resultado único para zeros à esquerda/finais, expoentes ou valores extremos.
- **Limites de campos ausentes:** F01 promete campos internos e públicos
  obrigatórios e limitados, porém card/spec/plano não fixam os limites nem a
  política de whitespace. Esses valores afetam schema, validação, i18n e API e
  precisam estar aprovados antes da execução, não escolhidos pelo executor.
- **Ownership sobreposto:** F02 possui `src/app/admin/products/**`, enquanto F03
  possui `src/app/admin/*product*`; este último também pode abranger o diretório
  `products`, apesar de F03 declarar escopo apenas de UI. Separe caminhos exatos
  e proíba F03 de editar as rotas de mutação para que os contratos efêmeros e o
  scope check sejam inequívocos.

- [x] Feito
