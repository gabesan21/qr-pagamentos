# 3.2.1-product-schema-and-admin-crud — Plan

## Objective and expected result

- **Objective:** persist bilingual products and provide active administrators with complete create, read, update, deactivate/reactivate, and delete controls without weakening the existing authorization, exact-decimal, i18n, or design-system boundaries.
- **Observable result:** `/admin` shows localized product creation and management states; saved products retain their exact positive price, both public-language variants, and active status, while stale edits fail with an opaque conflict and protected requests expose only empty `401`/`403` outcomes.

## Strategy

Add one forward-only product model and migration, then place product rules behind a cohesive server service consumed by native admin POST routes and the existing server-rendered admin page. Persist prices as strings matching the spec's canonical positive dot-decimal grammar: at most 12 integer digits and 6 fractional digits (18 total), no leading integer or trailing fractional zero, and no sign, exponent, comma, grouping, or whitespace; reject zero and overflow rather than trimming, normalizing, or rounding. This deliberately narrows the existing Nautt exact-decimal lexical boundary while retaining its string-only/no-`Number` invariant. Format the exact string for the persisted `pt-BR` or `en` locale only at presentation. Use an explicit persisted version for compare-and-swap updates/deletes so concurrent edits cannot silently overwrite each other.

Validate and persist trimmed text with spec-owned limits measured in Unicode code points: internal name 128, each public title 160, and each public description 2,000. Reject whitespace-only values and CR/LF in the single-line name/title fields; preserve internal whitespace and description line breaks.

Before composing the product surface, extend the owned Radix/nova inventory with the official shadcn `Textarea` primitive required by the approved multiline-description contract. Use the repository-pinned `shadcn@4.13.0` through `pnpm exec shadcn`, in this order: `info --json`, `docs textarea`, `add textarea --dry-run`, then `add textarea`; never fetch or copy raw registry source. Add it to the deterministic specimen and inventory evidence, then consume it from the admin surface without adapting it or replacing it with `Input`/raw `<textarea>`.

Extend the established PIX-ledger admin composition with that owned inventory and the semantic tokens already documented in `DESIGN.md`. Cover default/populated, pending/disabled, empty, error/recovery, hover/focus, and destructive-delete confirmation without adding a page-local visual variant. The catalog spec already owns the durable behavior; adding `Textarea` implements the approved contract and does not change the product spec. Record concrete delivered invariants and the implemented slice there only at closeout, without broadening into the public product read model reserved for 3.2.2.

## Dependencies, risks, and abort conditions

- **Task dependencies:** none; implementation assumes the integrated 3.1.1/3.1.2 admin catalog and design-system baseline already present on `develop`.
- **Exactness risk:** Prisma/UI coercion could round or normalize a price — mitigate with boundary tests for the exact canonical grammar, 18/6 precision/scale limits, zero, leading/trailing zeros, exponent, separators, and whitespace plus matching database checks; no `Number` conversion.
- **Text-boundary risk:** UI, service, and database could disagree on trimming or length — validate the trimmed Unicode-code-point length and single-line restrictions at the service boundary and enforce compatible database bounds.
- **Lost-update risk:** ordinary ID-only updates can overwrite a newer edit — require version-matched writes/deletes and a generic localized conflict outcome.
- **UI drift risk:** the growing admin page could introduce raw styles or a duplicate primitive — retain semantic tokens, owned components, source checks, and fresh responsive evidence.
- **Registry drift risk:** CLI output could target unexpected files or no longer match the configured Radix/nova contract — require pinned `pnpm exec shadcn` info/docs/dry-run evidence and abort before `add` if the preview is incompatible; never substitute a hand-authored or raw-fetched control.
- **Evidence-secret risk:** omitting the Compose-required key prevents the browser gate from starting, while a reusable or logged value would violate secret hygiene — generate a fresh valid disposable value inside the runner's temporary lifecycle, stage it with restricted permissions, include it in failure-log redaction, and rely on unconditional cleanup.
- **Abort if:** the approved 18/6 price contract or text limits cannot be represented consistently at the service and database boundaries, or a required product field/payment-link relationship changes — record `RECON NEEDED`, set `blocked: true`, and return to 002 rather than guessing or implementing Phase 3.2.2/3.3 behavior.
- **Abort F03a if:** `components.json`, installed CLI version, registry docs, or dry-run output conflicts with the expected single owned Textarea source; report `BLOCKED` without editing admin UI or replacing multiline descriptions with an incompatible control.
