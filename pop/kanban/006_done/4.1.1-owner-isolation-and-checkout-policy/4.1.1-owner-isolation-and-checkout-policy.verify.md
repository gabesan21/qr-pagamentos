## Findings

- **B1 — blocker — `src/auth/payment-link.test.ts:46-50,147-150`:** The test-store `deactivate(_ownerId, _id)` ignores both authorization inputs. The test first deactivates as the legitimate owner; the second-owner call then fails only because the shared `active` flag is already false. It would still pass if production `updateMany` dropped its `{ ownerId }` predicate, so it does not satisfy AC2's required two-owner mutation proof. Make the fixture retain links per owner and reject a foreign actor before the owner revokes, then assert the foreign link remains active; include the complementary successful owner mutation.
- **N1 — nit — `src/auth/payment-link.test.ts:4,46`:** The new test leaves an unused `ForbiddenError` import and unused mock parameters, producing the three lint warnings reported by `pnpm check`.

## Round 2 — 2026-07-21

### Scope and B1 regression proof

- **B1 resolved:** `src/auth/payment-link.test.ts` now stores independent links and active state per link, and its `deactivate(ownerId, id)` lookup requires both values. The foreign actor attempts to revoke the owner's still-active link first and must receive `PaymentLinkConflictError`; both links are then asserted active. The rightful owner then revokes only its own link, while the foreign link remains active.
- **Regression sensitivity:** removing the fixture's owner match makes the foreign attempt deactivate the owner's link and therefore fail its expected rejection. Independently, the production store retains the separate database predicate `where: { id, ownerId, active: true }` in `src/auth/payment-link.ts`; source review confirms the service supplies only `actor.id`.
- **No adverse change found:** the correction is test-only, removes the former unused import/mock-parameter lint issue (N1), and leaves the public payment-link resolver, checkout/order paths, and Nautt integration outside the diff.
