## Route

Before execution, the orchestrator creates `task/1.2.1-review-clean-code-baseline` from local `develop` after synchronizing it with `origin/develop`. Execution records that exact commit as `TASK_BASE` in g01 Notes and proves it is an ancestor of `HEAD`; all later diff checks use `TASK_BASE...HEAD`, never `HEAD^` or the moving `develop` ref.

Execution starts fail-closed. If the host does not already report the exact pins, run the complete g01 recovery block: build the Dockerfile `toolchain` target with `--pull=false`, create a temporary container, copy its Node binary and pnpm installation to `/tmp/qr-pagamentos-1.2.1-toolchain`, prepend that directory to `PATH`, and observe `v24.18.0` / `11.13.0`. This keeps database/container harnesses on the host so their Docker calls work. If the cached build or extraction fails, block rather than download or guess. Then run `pnpm install --frozen-lockfile` unconditionally, confirm dependency memories, and run the cheap gates before editing. Capture failures as baseline evidence; a failure is not automatically permission to rewrite behavior.

Then review two bounded fronts in parallel. The application/database front owns only the ledger in g02 Notes; the deployment front owns only the ledger in g03 Notes, so parallel writers never share an artifact. For each candidate issue, record the evidence and classify it as: no issue; local safe remediation; or larger debt tracked with context, impact, and next step. Remediate only confirmed local issues, one cohesive transformation at a time, protecting current behavior with an existing or newly added characterization test. After both groups finish, g04 copies their finalized rows in fixed g02-then-g03 order into its consolidated ledger before judging the combined diff.

Finish by reviewing the diff as one behavior-preserving baseline change, walking all applicable DOX contracts, updating only contracts whose purpose/structure/flow changed, synchronizing the mounted spec notes if reality diverged, and running every acceptance gate. Commit before clean-clone container scenarios so their clean-HEAD guard is meaningful.

Discarded alternative: broad stylistic rewrites or new abstractions are outside scope because they lack demonstrated maintenance value and would obscure behavioral preservation.

## Forks

- If an existing gate fails before edits in g01 → preserve its output, isolate whether it is environment-only or a Phase 1.1 defect, and remediate only a confirmed in-scope defect in the owning review group.
- If a candidate issue has no protecting test but behavior is observable → add the smallest characterization test first, show it passes before refactoring, then make one structural transformation.
- If Prisma schema or bootstrap behavior needs a fix → follow `prisma/AGENTS.md`; create a new migration only for a real schema change and never edit the applied foundation migration.
- If a change touches installer commands, validation, deployment, health waiting, or uninstall flags → run `install/test.sh` in that movement before continuing.
- If a deployment change affects bootstrap/migration/runtime routing → run both the relevant static contract and the smallest matching disposable container scenario before the final full matrix.
- If no confirmed issue exists in a reviewed surface → record the evidence and make no code change there.

## Abort conditions

- Exact Node `24.18.0` and pnpm `11.13.0` cannot be made available, or frozen dependency installation fails for a non-transient reason.
- The task branch was not created from a synchronized local `develop`, `develop` differs from local `origin/develop`, or the recorded `TASK_BASE` is not an ancestor of the task branch.
- A proposed remediation changes a requirement in the mounted spec, introduces identity/access behavior, or needs a product decision.
- A finding requires broad refactoring, an unplanned migration-history edit, weakened role/secret/container boundaries, or another task-sized change.
- A relevant AGENTS contract is missing/contradictory, a gate failure cannot be attributed, or actual behavior diverges materially from the approved plan/spec.
- Disposable Docker prerequisites/resources cannot be isolated safely; never reuse operator secrets, volumes, or projects.
