```prisma

  userId          String   @id @map("user_id") @db.Uuid
  encryptedApiKey String   @map("encrypted_api_key") @db.Text

  user            User     @relation(fields: [userId], references: [id], onDelete: Cascade)

  @@map("nautt_credential")
  @@schema("app")
}
```

2. Add `nauttCredential NauttCredential?` relation to the `User` model.
3. Create a new versioned migration that:
   - Creates `app.nautt_credential` with `user_id` PK, `encrypted_api_key TEXT NOT NULL`, timestamps, and FK to `app.user(id) ON DELETE CASCADE`.
   - Grants `SELECT, INSERT, UPDATE, DELETE` on `app.nautt_credential` to `qr_runtime` (DML only; no DDL).
   - Keeps constraint names deterministic.

## Implementation steps

1. **Cryptography primitive**
   - Implement `src/lib/nautt-crypto.ts` using Node `crypto`:
     - `encrypt(plaintext: string, key: Buffer): string` returns base64url(`iv`) + ":" + base64url(`ciphertext`) + ":" + base64url(`authTag`) for AES-256-GCM.
     - `decrypt(ciphertext: string, key: Buffer): string` parses the composite, verifies tag, returns plaintext.
     - Random 12-byte IV per encryption; never reuse IVs.
     - Reject keys that are not exactly 32 bytes.
     - Fail closed on malformed ciphertexts, wrong tags, or missing key.
   - Load the key from `NAUTT_ENCRYPTION_KEY` env var (base64url 32-byte value). In production the runtime wrapper injects it; in local dev it comes from `.env`.

2. **Credential service**
   - Define `NauttCredentialStore` with `upsert(userId, encryptedApiKey)`, `find(userId)`, and `exists(userId)`.
   - Implement `createNauttCredentialService(store, crypto)`:
     - `save(actor, targetUserId, apiKey)` â€” require authenticated active principal; allow only if `actor.id === targetUserId` or `actor.role === "ADMIN"`. Encrypt the key inside the service and upsert. Trim whitespace and reject empty keys.
     - `getRedacted(actor, targetUserId)` â€” same authorization; returns `{ hasCredential: boolean; updatedAt: Date | null }`.
     - `getDecryptedApiKey(targetUserId)` â€” **server-only** helper for downstream integration modules (pricing, orders, webhooks); not exposed through any HTTP response.
     - Throw distinct `ForbiddenError`, `NauttCredentialNotFoundError`, `NauttCredentialValidationError`.
   - Never log the plaintext or encrypted key; use redaction helpers if needed for error paths.

3. **Admin mutation route**
   - Add `POST /admin/users/[id]/nautt-credentials` that:
     - Re-authorizes via `requireAdminFromCookie`.
     - Reads `apiKey` from `FormData`.
     - Calls `getNauttCredentialService().save(actor, targetUserId, apiKey)`.
     - Returns empty `401`/`403` via `protectedMutationResponse`, or `303` redirect to `/admin?success=nautt-credentials` / `/admin?error=nautt-credentials-failed`.
   - Keep the route body small; authorization logic lives in the service so 2.1.2 can reuse it.

4. **Secret pipeline**
   - Add `NAUTT_ENCRYPTION_KEY` to `.env.example` and `install/.env.example` with a comment marking it as required, 32-byte base64url.
   - Update `install/install.sh` to accept `NAUTT_ENCRYPTION_KEY`; if absent, generate `crypto.randomBytes(32).toString("base64url")` and warn the operator to back it up. Validate provided keys decode to 32 bytes.
   - Stage the key into `.container-secrets/nautt_encryption_key` (mode `0400`, uid `1000`).
   - Add the secret to `compose.yaml` and mount it only on the `app` service.
   - Update `container/runtime.mjs` to read `/run/secrets/nautt_encryption_key` and inject it as `NAUTT_ENCRYPTION_KEY` into the spawned app process, alongside `DATABASE_URL`.

5. **Tests**
   - Unit tests for `nautt-crypto.ts`: round-trip, wrong key fails, tampered ciphertext fails, missing key throws.
   - Unit tests for `nautt-credential.ts` using an in-memory store:
     - Owner can save and read their own redacted credential.
     - Admin can save/read for another user.
     - Non-owner non-admin user is rejected with `ForbiddenError`.
     - Disabled actor is rejected.
     - Saving twice replaces the previous value (same `userId`, new `encryptedApiKey`, updated `updatedAt`).
     - `getDecryptedApiKey` returns the original plaintext only for the owner.
     - Redacted output never contains the key.
   - Route tests for `/admin/users/[id]/nautt-credentials`:
     - Empty `401`/`403` responses.
     - Admin can POST a key; target user ID comes from URL, not form.
     - Opaque failure redirect without disclosing the target or key.
   - Run `pnpm db:generate`, `pnpm db:test`, `pnpm check`.

## Risk/red-team notes

- **Key loss = data loss**: the encryption key is not stored in the database. If the operator loses it, every persisted Nautt API key is irrecoverable. The install script must warn about backup and never print the generated key to logs.
- **Key exposure**: `NAUTT_ENCRYPTION_KEY` must never be committed, passed to clients, or included in error responses. Verify with grep and route tests.
- **Authorization bypass**: the service must re-check the actor on every call; route-level admin guard is not enough for the owner-scoped service because 2.1.2 will add a user-facing route.
- **IV reuse**: crypto module must generate a fresh IV per encryption call; add a test that two encryptions of the same plaintext produce different ciphertexts.
- **Timing/oracle**: decryption failures must not expose whether a record exists or whether the ciphertext was malformed; return generic errors.
- **Scope creep**: this task does not call Nautt, validate the key against the wallet endpoint, register webhooks, or render the onboarding UI. Keep those in 2.1.2 and 2.1.3.
