# 3.1.2 — Server-side catalog UUID validation

## Objective

Harden server-side validation and authorization for all catalog mutations so that only active administrators can create, update, or toggle currency-pair and payment-method records, malformed UUIDs are rejected before persistence, and unauthorized or invalid requests never disclose target records or input values.

## Affected files

- `src/auth/nautt-catalog.ts` — service-layer UUID/label validation and admin authorization.
- `src/app/admin/catalog/currency-pairs/route.ts` — create currency pair.
- `src/app/admin/catalog/payment-methods/route.ts` — create payment method.
- `src/app/admin/catalog/currency-pairs/[id]/route.ts` — update/toggle currency pair.
- `src/app/admin/catalog/payment-methods/[id]/route.ts` — update/toggle payment method.
- `src/auth/nautt-catalog.test.ts` — service validation tests.
- `src/app/admin/catalog/**/*.test.ts` — route authorization and validation tests.
- `src/i18n/dictionaries/pt-BR.ts` and `src/i18n/dictionaries/en.ts` — only if new user-visible validation copy is required.

## Strategy

1. **Keep validation in the service.** The canonical UUID regex and label checks live in `src/auth/nautt-catalog.ts`; routes pass raw form values through and let the service reject malformed input with `NauttCatalogValidationError`.
2. **Authorize first, parse second.** Every catalog POST must call `requireAdminFromCookie` before reading the request body and must return only empty `401`/`403` responses via `protectedMutationResponse` for unauthenticated or non-admin callers.
3. **Opaque failure handling.** Route catch blocks must redirect to the existing catalog error query parameter (`?error=catalog-create-failed` / `?error=catalog-change-failed`) for validation, not-found, or persistence failures. Never echo submitted UUIDs, labels, or record existence back to the client.
4. **Close the test gap.** Current route tests mock the service and therefore do not exercise real UUID validation. Add route-level tests that drive the actual service (or a real in-memory store) and assert that malformed UUIDs result in the opaque error redirect while valid UUIDs succeed.
5. **Extend service test coverage.** Cover uppercase UUID normalization, empty/missing/null UUID fields, and invalid identifier values on update/toggle paths.
6. **Localize only UI-facing outcomes.** Internal service error messages remain English; any user-visible validation copy is sourced from the existing dictionary keys (`adminCatalogCreateFailed`, `adminCatalogChangeFailed`) and their `pt-BR`/`en` equivalents.

## Acceptance criteria

- [ ] Unauthenticated and non-admin callers receive empty `401`/`403` responses from all four catalog POST routes.
- [ ] Creating a currency pair or payment method with a malformed, empty, or missing UUID is rejected and redirected opaquely; no value or record detail is leaked.
- [ ] Updating/toggling a record with an invalid `[id]` or malformed body is rejected and redirected opaquely.
- [ ] Valid canonical UUIDs are accepted; uppercase UUIDs are normalized to lowercase before persistence.
- [ ] `pnpm lint`, `pnpm typecheck`, and `pnpm test` pass.

## Risks

- **Mock-heavy route tests may hide real validation behavior.** Mitigation: add tests that exercise the service-to-route path with real validation logic.
- **Internal validation messages could be exposed if catch blocks change.** Mitigation: keep service errors internal; routes always redirect to dictionary-backed error states.
- **Tightening UUID validation could break legitimate but previously tolerated inputs.** Mitigation: accept only canonical UUIDs per spec; reject everything else.
