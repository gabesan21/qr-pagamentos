# 4.2.2-public-checkout-experience — Plan

## Objective, boundary, and proposed durable contract

Deliver the unlocalized, sessionless `/pay/[identifier]` page for one exact payment-link identifier. Its server-only projection is exactly `{ product: { title, description, price }, checkoutPolicy }` with the five lifecycle-specified policy values; the browser supplies only the policy-exact customer object and a fresh opaque idempotency key to `POST /api/payment-links/[identifier]/checkout`. It must never call an owner route or derive/submit owner, price, pair, provider identity, credential, link/order identity, or status; it retains retry key, customer input, capability, and PIX payload only in live component memory, never in a URL, cookie, storage, analytics, console, error, or cache.

The lifecycle contract pins `POST /api/payment-links/[identifier]/checkout/status`: it accepts only `{ statusCapability }`, validates its versioned re-derived verifier plus immutable attempt binding/future expiry/unset revocation, and returns only the closed redacted payment DTO and state vocabulary specified there. Every syntactically valid status loser is the same empty no-store `404`; the page consumes the existing public resolver/checkout contracts, not a new broad client read. The server remains authoritative for link and capability expiry; the UI shows one generic unavailable/expired recovery and never exposes a countdown or distinguishes its cause.

Each user intent generates one high-entropy key and freezes its exact submitted payload; a disabled submit and same-key manual retry may replay it, while any edit/new attempt generates a new key. `201` shows returned QR/copy-paste data; `202` shows noncommittal creation/indeterminate feedback. Copy runs only on an explicit user gesture and announces success/failure without logging the value. QR/copy absence is a clear waiting/retry-status state, not fabricated payment data. Polling sends only the in-memory bearer, has one abortable non-overlapping timer, pauses while hidden/unmounts, stops on terminal or unavailable outcome, and may retry the status read—not checkout POST—through an explicit bounded/backoff policy.

## Execution fronts

### F01 — redacted public projection and capability-status boundary

- **Owns:** lifecycle-spec amendment, server-only checkout presentation resolver, capability status route/DTO, and focused route/resolver tests. **May read:** root and checkout DOX, linked lifecycle/catalog specs, 4.2.1 memory. **Must not edit:** page/components/dictionaries, owner/admin surfaces, provider dispatch/recovery/webhooks, or generated sources. **Depends on:** 4.2.1. **Criteria:** AC1–AC3; review root and `src/checkout/AGENTS.md` at closeout.

### F02 — bilingual public page and safe interaction state machine

- **Owns:** the approved public route, client form/state machine, dictionary keys, focused UI tests, and `DESIGN.md`/DOX review if its inventory or scope changes. **May read:** F01’s committed redacted contracts and existing owned primitives. **Must not edit:** checkout/provider persistence or F01 API semantics. **Depends on:** committed F01. **Criteria:** AC4–AC6; use `Field`/native controls, `Card`, `Alert`, `Badge`, `Separator`, `Skeleton`, `Spinner`, and `Button` variants before any new primitive. F01 → F02 is sequential; no write-safe parallelism exists.
