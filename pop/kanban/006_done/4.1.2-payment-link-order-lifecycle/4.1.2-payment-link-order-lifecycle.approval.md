# Approval — [[4.1.2-payment-link-order-lifecycle]]

## Round 1 — 2026-07-21

### Decision brief

- **Delivery:** Owner-bound payment-link order persistence, policy-exact private customer snapshots, closed local lifecycle states, expiry-gated creation, and an atomic one-winner settlement claim for single-use links.
- **Relations:** An order references the same owner/link/product; it snapshots the payable price and pair UUIDs; a provider order is nullable before 4.2.1 and can attach to only one same-owner order. Existing unattached provider orders remain valid.
- **Concurrency:** Pending/failed/cancelled/expired/indeterminate orders never consume a link. A short transaction records one immutable settlement claim only after authoritative reconciliation; refunds never release it, while reusable links remain multi-settlement.
- **Exclusions:** No checkout/status UI or routes, customer submission endpoint, provider request/decryption/quote, Nautt retry, polling/webhook/recovery behavior change, protected order view, or public PII/status exposure.
- **Topology and proof:** `F01 persistence → F02 lifecycle`; tests must be regression-sensitive for expiry and two confirmed contenders. Run focused tests, database generation/contract checks, `pnpm check`, and diff check. Docker, `db:test`, and browser/runtime are authorized deferred evidence.
- **Specs:** Existing checkout/order, Nautt, and catalog/link specs already define the owner, privacy, expiry, settlement, and authoritative-reconciliation boundaries. No spec amendment is proposed; catalog/link behavior is unchanged.
- **Plan:** [[4.1.2-payment-link-order-lifecycle.plan]] — *follow for the complete execution brief*.

### Response from reviewer (yolo)

- **Decision:** Returned to 002 (round 1).
- **Blocking — snapshot contract:** “structured address snapshot” is undefined. Specify its persisted shape, mandatory/optional members, per-field normalization and bounds, and the exact nullability/policy tuple. Without that, the claimed database proof that all five policies permit *exactly* their fields (AC1/AC3) is not implementable or reviewable, and an arbitrary JSON address could silently expand the PII contract.
- **Blocking — settlement fence:** Define the lifecycle-store settlement input and its one-transaction conditional predicates: the attached same-owner provider order, its observed reconciliation version and authoritative confirmed status, the eligible local state(s), and the loser/idempotent outcomes. The current plan says a version check occurs but does not state what is checked in the claim-and-confirm transaction; it could otherwise confirm `CREATED`/a stale or unattached order, or race a later provider reconciliation. This must also state how `processing`/`paid`/`finished` map to the closed local states and which local states are terminal before F02 can write regression-sensitive tests.

## Round 2 — 2026-07-21

### Revised decision brief

- **Draft contract requiring renewed 003:** [[specs/checkout-and-order-lifecycle|Checkout and order lifecycle]] is now `draft`. It fixes `CustomerSnapshotV1`, including the exact five policy tuples and a fixed Brazil-only address V1: required street, number, district, city, UF, CEP, and `BR`; optional complement; NFC/trimmed bounded text; normalized valid CPF/CEP; closed UF; and no JSON/unknown fields.
- **PII boundary:** Snapshot fields are private fixed persistence only—never public, provider-facing, logged, or URL data. This task adds neither a customer-submission route nor an owner/admin projection.
- **Fenced settlement seam:** `SettlementMapV1` maps only the existing authoritative Nautt evidence: `new → PENDING`; `processing`/`paid`/`finished → CONFIRMED`; `rejected → REJECTED`; `canceled → CANCELLED`; `expired → EXPIRED`; `refunded → REFUNDED`. Unknown provider statuses and all event/history values are no-ops. Only the three mapped authoritative success statuses can enter `CONFIRMED`.
- **Conditional atomicity:** Future reconciliation supplies the attached same-owner provider order, UUID, observed provider reconciliation version, and mapped authoritative status. One transaction also matches local version/current eligible state, creates the single-use claim before confirmation, and fences stale, duplicate, cross-owner, unattached, ineligible, and losing inputs as opaque no-ops. `CREATED` cannot settle; `PENDING`/`INDETERMINATE` are eligible before a terminal mapped result, and only `CONFIRMED → REFUNDED` is later permitted.
- **Scope:** Task 4.1.2 creates only persistence and the local reconciliation seam/tests. It makes no provider call and does not change polling, webhook, or recovery execution; the later checkout orchestration task supplies the existing authoritative reads.
- **Topology and proof:** `F01 persistence → F02 lifecycle`; test exact snapshot normalization/tuple limits and every fenced success/loser path, then run focused tests, database generation/contract checks, `pnpm check`, and diff check. Docker, `db:test`, and browser/runtime remain authorized deferred evidence.
