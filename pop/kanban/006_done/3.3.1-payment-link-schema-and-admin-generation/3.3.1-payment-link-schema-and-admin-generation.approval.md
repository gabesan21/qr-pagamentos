## Round 1 — 2026-07-20

### Resposta do revisor (yolo)

devolvida pelo revisor (yolo) — 2026-07-20

The plan is precise about the schema, opaque identifier, expiry and revocation
boundaries, authorization, i18n, UI ownership, price-string handling, and the
exclusion of public resolution, checkout, order consumption, expiry jobs, and
provider work. Its three fronts are sequential and their declared write
ownership is non-overlapping.

It cannot be approved yet because its active-dependency concurrency claim is
not executable against the current catalog implementation. The existing catalog
deactivation route calls `setCurrencyPairActive`, whose Prisma store performs a
plain `update` with no row lock or shared serializable protocol. A creation
transaction that merely reads `active = true` at `Serializable` isolation can
therefore serialize before that independent deactivation while physically
committing after it; serializability alone does not establish the plan's stated
"active at the commit boundary" guarantee. The same concern applies to product
deactivation.

Return to 002 with one concrete, ownership-compatible protocol that makes the
creation and the existing product/pair deactivation paths contend on the same
rows (for example, transaction-scoped row locking followed by the active check,
or an equivalent conditional-write contract). State the exact affected owner(s)
and dependency ordering, and make AC3's focused test deterministically prove
that a deactivation racing the create cannot leave a newly committed link whose
dependency was already inactive at the create commit boundary. Preserve the
later-deactivation rule: after a successful create commits, a subsequent
deactivation may leave the existing FK intact without rewriting the link.

## Round 2 — 2026-07-20

### Resposta do revisor (yolo)

aprovado por revisor independente (yolo) — 2026-07-20

The revision supplies the missing shared protocol without changing ownership of
the already-delivered deactivation paths. The migration-owned `BEFORE INSERT`
trigger takes transaction-scoped `FOR SHARE` locks in the single global order
`Product` then `CatalogCurrencyPair`, and PostgreSQL's ordinary non-key
`UPDATE` lock used by both current active-state mutations conflicts with that
share lock. Therefore, across the application's independent creation and
deactivation transactions, a deactivation that commits first is re-read as
inactive after the insert waits and is rejected with the specified named
`23514`; an insert that acquires both locks first commits before the waiting
deactivation can proceed. The fixed two-row order also avoids a lock cycle:
creation transactions use the same order, while each current deactivation
updates only its one dependency row.

AC3 and the abort condition now require a proportionate deterministic proof:
separate `qr_runtime` clients hold each side of the race, await actual lock
waits through `pg_stat_activity`, and prove both commit orders, the named
constraint error, and the absence of a failed insert. This is not a timing or
serializable-snapshot assertion. The remaining persistence, authorization,
opaque-outcome, exact-price, UI/i18n, and scope-exclusion contracts are
specific, owned by sequential fronts, and have executable 005 criteria.
