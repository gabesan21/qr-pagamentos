# 3.3.1-payment-link-schema-and-admin-generation — Plan

## Objective and expected result

Deliver the administrator-only foundation for creating and revoking payment links. A link persists an application-generated opaque identifier, exactly one product and catalog currency-pair reference, `SINGLE_USE` or `REUSABLE` type, nullable expiry, active state, and creation metadata. Creation atomically accepts only dependencies active at its commit boundary, mutually serialized with product/pair deactivation; it preserves the product's exact canonical price string without parsing, formatting, copying, or converting it through `Number`.

An authenticated active administrator can generate and inspect links in `/admin`, then explicitly deactivate them as a manual revocation. New links are active; revocation is not a reactivation control. Every admin read and POST independently re-authorizes the cookie principal. Protected failures remain empty `401`/`403`; all other mutation failures redirect only to existing localized opaque recovery states. This task creates neither a sessionless link route nor checkout, product public-read use, provider request, order, consumption transition, or automatic expiry/write-side inactivation.

## Strategy and contracts

1. Add one `PaymentLink` Prisma model and forward migration. It has a UUID primary key, a unique `identifier` of bounded URL-safe ASCII text, `linkType`, nullable `expiresAt`, `active`, `createdAt`, and restrictive foreign keys to `Product` and `CatalogCurrencyPair`. SQL supplies named checks for the two link types and the URL-safe identifier grammar, a named unique constraint, indexes needed for identifier lookup and administration, least-privilege runtime DML, and one named `BEFORE INSERT` trigger. Existing products/pairs are never modified. `Product` and `CatalogCurrencyPair` expose the inverse relation only.
2. The server-only service owns generation and one-way manual revocation. It creates a fresh `randomBytes(...).toString("base64url")` identifier on the server (fixed 24-character token), always persists a new link as active, never accepts an identifier or active value from a form, and retries only a bounded unique-collision insert before returning its generic service failure. The service explicitly validates a canonical UUID product/pair reference, closed link type, and an optional UTC `datetime-local` value (`YYYY-MM-DDTHH:mm`, interpreted as UTC); blank expiry becomes `null`, malformed/non-round-trippable or non-future values fail. This is an administrator input convention, not a public-link URL or resolution contract.
3. The migration-owned `payment_link_require_active_dependencies` trigger is the shared creation/deactivation protocol. Before every link insert it locks the referenced `Product`, then `CatalogCurrencyPair`, with `SELECT ... FOR SHARE`, checks both rows exist and are active, and raises `23514` with `payment_link_product_active` or `payment_link_currency_pair_active` for a missing/inactive dependency. Those transaction-scoped share locks conflict with the existing ordinary `UPDATE` statements used by both product and currency-pair activation changes, so no route/service change is needed: a deactivation that commits first makes the insert fail; an insert that acquires both locks first commits while both dependencies remain active and only then lets a later deactivation proceed. The service maps either trigger failure to its existing generic opaque dependency failure; serializable isolation may be used for its surrounding work but is not the correctness mechanism. Later deactivation does not rewrite or remove a successfully committed link; future public resolution will decide availability without treating an inactive catalog pair as a broken foreign key. Foreign-key restriction keeps referenced records from being deleted silently.
4. Extend the existing admin server page and surface with a payment-link generation section. It uses only the owned `Card`, `Field`/`Input`, `NativeSelect`, `Alert`, `Badge`, `Separator`, `Button`, and pending `Spinner` conventions and existing semantic CSS tokens. Selects contain only current active products/pairs; an explicit localized empty/recovery state replaces the form when either prerequisite is absent. The populated list shows the admin-safe identifier, type, selected labels, localized exact-price presentation through the existing string formatter, expiry, and active status. It renders a confirmation before one-way manual deactivation. It must not render a public `href`, public product fields beyond the administrator's existing scope, a new primitive, raw CSS/token value, or a link resolver.
5. Add both dictionary languages in lockstep and extend the existing opaque notice mapping. No change to `DESIGN.md` or `src/components/ui/` is expected: the current PIX-ledger tone, token contract, component inventory, and state matrix already cover this form/list. If execution discovers a genuine inventory/state-contract change, stop and update the applicable contract in the same front; do not hand-roll a primitive.

## Risks, limits, and abort conditions

- **Dependency race:** list-time active choices can become inactive before POST. The migration trigger must take product then pair `FOR SHARE` locks, check active state while locked, and hold them through the insert transaction commit; it must use `payment_link_product_active`/`payment_link_currency_pair_active` `23514` failures that F02 maps to one opaque outcome. `pnpm db:test` must use separate runtime connections and `pg_stat_activity` lock-wait observation, not timing or serializable-read assumptions: first prove each existing product/pair deactivation `UPDATE` waits behind an uncommitted creation and runs only after it commits; then let that deactivation acquire its update lock and remain uncommitted, observe the insert wait, commit deactivation, and prove the insert fails with the matching named constraint and leaves no link.
- **Identifier weakness or leakage:** use CSPRNG base64url output, database uniqueness, bounded collision handling, and no form-supplied identifier. Identifier visibility is restricted to the administrator list; no public path, route, or link element belongs to this task.
- **Lifecycle expansion:** `SINGLE_USE` persistence does not consume a link. Do not add order fields, provider calls, automatic expiry jobs, or the 3.3.2 resolver; a later successful order flow will own the atomic consume transition.
- **Exact-money regression:** the link stores references, not a numeric/duplicated price. The UI may invoke the existing exact string formatter only for display; no service/database/route path may parse or convert price with `Number`.
- **Authorization disclosure:** authorize before reading form data, service dispatch, or target lookup. Never distinguish malformed, missing, inactive, stale, duplicate, or locked dependencies in redirects or response bodies.
- **UI/token drift:** use the existing inventory and semantic tokens. Do not run a shadcn `add` unless execution finds a real missing primitive; then obtain pinned CLI `info`, component `docs`, and `add --dry-run` evidence first. No new token values are planned.
- **Abort and return to 002** if the forward migration cannot enforce or deterministically prove the trigger's shared row-lock protocol and named dependency failures with the existing deactivation `UPDATE`s, if FK/type/identifier invariants cannot be reliably tested, or if a requirement needs public resolution/checkout/order consumption. Record `RECON NEEDED`; do not implement a substitute boundary.
