# Plan — [[5.3.2-public-endpoint-rate-limiting]]

- **Stage:** 002_planning · **Owner:** planning agent

## Objective and expected outcome

Add one process-local, bounded rate-limit decision point for every existing public payment-link API endpoint. A request over its configured budget receives only an empty, `Cache-Control: no-store` `429`; it must not reach JSON parsing, locale negotiation, a public-link read, checkout/status service, provider I/O, or persistence. App Router document pages are not route handlers and are explicitly outside this task.

## Strategy

Create a server-only, dependency-injectable token-bucket limiter (monotonic injected clock) with a small closed surface enum, per-surface budgets, and a hard maximum number of entries. Its key is `surface + hash(normalized trusted client address)`: never include an identifier, slug, status capability, retry key, cookie, account, request body, or raw address. The app is bound to loopback, so the only candidate client address is one canonical IP literal from `X-Forwarded-For`, **provided the TLS reverse proxy overwrites that header**; chains, missing, malformed, overlong, or noncanonical values use one shared anonymous key and never bypass the limiter. No endpoint exposes its key, remaining quota, reset time, or limiter state.

One synchronous `allow()` decision mutates its `Map` before an `await`, so requests in one Node process cannot race between check and debit. Each entry has only token count and last-refill/touch time; lazy expiry removes idle entries, capacity is capped, and a full fresh map denies an unseen key instead of allocating. There are no timers, persistence, cross-process guarantees, or global/serverless claims: the documented single-instance topology is the scope.

## Route inventory and ownership

| Surface | Owner at 004 | Budget/key class | Integration point |
|---|---|---|---|
| `GET /api/payment-links/[identifier]` | `src/app/api/payment-links/[identifier]/route.ts` | public-link read | before locale negotiation/service read |
| `POST /api/payment-links/[identifier]/checkout` | `src/app/api/payment-links/[identifier]/checkout/route.ts` | checkout submit | before `request.json()`/checkout service |
| `POST /api/payment-links/[identifier]/checkout/status` | `src/app/api/payment-links/[identifier]/checkout/status/route.ts` | payment-status poll | before `request.json()`/status service |

`/pay/[identifier]` and `/store/[slug]` are explicitly excluded: their App Router Server Component page owners cannot return an HTTP `Response`, and this task must not add forbidden middleware/proxy or alter their document contracts. `/api/health`, `/api/nautt/webhooks`, authenticated routes, and `[lang]/*` stubs are also outside the closed public payment-link inventory.

The shared implementation and its focused tests belong in a new server-only `src/security/public-rate-limit.ts`; it must have no React, Prisma, auth, checkout, provider, or logging dependency. `src/checkout/AGENTS.md` applies to the two checkout POST integrations. The root `AGENTS.md` must be updated with the public-rate-limit ownership/trust and opaque-outcome rule; no child contract is warranted for the new small security helper.

## Execution fronts

### F01 — Closed limiter and boundary characterization

- Implement the limiter plus unit tests for bucket refill/burst exhaustion, surface separation, address normalization/hash redaction, anonymous fallback, expiry, capacity-full denial, eviction/cleanup, and same-tick sequential decisions. Budgets and capacity are named constants, not environment inputs or request-controlled values.
- Characterize every public route above before integration; the only accepted result is `{ allowed: false }`, never an error/reason. A route adapter creates the identical empty/no-store `429` response.

### F02 — Public API route integration and durable contracts

- Apply the identical adapter at the three listed API entry points before parsing, locale negotiation, or service work, preserving all existing non-429 status/body mappings. Do not edit `/pay`, `/store`, `middleware.ts`, `proxy.ts`, cookies, origin guard, or add a public mutation.
- Update [[pop/specs/catalog-and-payment-links|Catalog and payment links]] for the public-link read and [[pop/specs/checkout-and-order-lifecycle|Checkout and order lifecycle]] for checkout/status with their respective opaque public `429` boundary and `no-store` invariant; no new spec is needed. Update `AGENTS.md` and the task memory/telemetry with the fixed API inventory plus single-instance and proxy-overwrite limit.

## Dependency, risks, and abort conditions

- This task has no dependency on 5.2.1: that task owns a document page, not a public API route handler. Do not recreate its projection or alter its redaction/UI contract. A later public storefront API endpoint must be added to a separately approved inventory and integration task.
- Do not add `middleware.ts` or `proxy.ts`: under the epoch decision, neither may supply document-page throttling. Do not silently substitute a `200`, `404`, redirect, or distinguishable page throttle for `429`.
- If deployment cannot establish a proxy that overwrites `X-Forwarded-For`, all untrusted/malformed forwarding remains on the shared anonymous key; do not trust a spoofable address or add distributed storage. If a multi-instance deployment is introduced, escalate the architecture/spec decision before claiming a global rate limit.
- Risks: NAT/shared anonymous clients can be conservatively throttled; process restart resets limits; capacity exhaustion can deny new keys. These are explicit limits of bounded single-instance protection, never reasons to retain unbounded keys or trusted secrets.
