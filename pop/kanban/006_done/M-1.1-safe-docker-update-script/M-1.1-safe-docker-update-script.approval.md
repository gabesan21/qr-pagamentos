# M-1.1-safe-docker-update-script — Approval

## Round 1 — 2026-07-22

### Resposta do crítico (yolo)

**Decision: return to 002 (return 1 of 2).**

Blocking reasons:

1. The existing-installation proof is weaker than the objective. Presence of the fixed `qr-pagamentos_postgres-data` name plus local secret files does not prove that the volume belongs to the compatible Compose installation. The plan must require and test fail-closed ownership/identity evidence (including rejection of a colliding or unlabeled volume) before `build` or `up`.
2. The acceptance evidence is mocked/static only. For this critical state-changing runtime command, add a disposable live install → update → re-run scenario that records the volume identity and Nautt source/staged key digests before the first update, then proves both remain identical while the one-shot/startup and exact health gates pass after each run. This is also the required runtime `re-run` criterion; it must never target operator resources.
3. The plan promises retention of previous-release metadata, and the runbook requires the pre-upgrade source revision, Compose/image identities, redacted configuration fingerprint, expected migrations, and backup reference, but no strategy or executable criterion preserves or verifies that rollback evidence before `build --pull` mutates release state. Define secret-free evidence and its verification, or narrow the promise and explicitly make the documented preflight a fail-closed prerequisite.

## Round 2 — 2026-07-22

### Resposta do crítico (yolo)

**Decision: approved.**

The revision resolves all round-one blockers with fail-closed Compose label/container/mount ownership proof, a uniquely isolated live install → update → re-run scenario that checks volume and Nautt-key invariants plus every startup gate, and protected atomic rollback evidence written before pull/build and retained on failure. The criteria are observable, cover the original request and linked durable contracts, contain no avoidable `(user)` work, and use proportionate deterministic, aggregate, and single live-runtime evidence.
