# M-1.1-safe-docker-update-script — Plan

## Refined objective

Add a dedicated operator command that updates an existing compatible `qr-pagamentos` Compose deployment without creating, replacing, printing, or deleting its database volume or protected secret sources. Before pulling or building, the command must fail closed unless it proves Compose ownership of the running database and volume, preserves secret-free rollback evidence, proves the deployed and source Nautt encryption keys agree, and receives a fresh backup reference.

## Strategy

- Introduce `install/update.sh` with the same prerequisite and redacted-output posture as the installer. Its explicit interface is `--env-file <path>`, `--backup-reference <non-secret reference>`, `--previous-release <non-secret source/release reference>`, optional `--evidence-dir <path>` (default ignored `.update-evidence/`), and `--dry-run`; missing rollback inputs are abort conditions for a real update.
- Load only the non-secret deployment settings needed by Compose, validate the existing `.install-secrets` and `.container-secrets` artifacts without rewriting them, and validate that the source/staged Nautt key is the same valid 32-byte base64url key. A populated `NAUTT_ENCRYPTION_KEY` in the environment file must also match; a blank value must never trigger generation.
- Before mutation, prove that `qr-pagamentos_postgres-data` has the expected Compose project and logical-volume labels and local driver, and that the existing `db` container has the expected Compose project/service labels and mounts that exact volume at `/var/lib/postgresql`. Reject missing, unlabeled, foreign, colliding, or inconsistently mounted resources before `build` or `up`.
- Atomically write mode-`0600` rollback evidence in a protected directory before `build --pull`: timestamp/run id, operator-supplied previous-release and backup references, target Git revision plus clean/dirty state, current Compose/container image identities, compatible volume identity/labels, a SHA-256 configuration fingerprint computed without persisting rendered values, and the expected migration filenames/digest. Failure to collect or persist the complete record aborts the update; neither evidence nor logs contain configuration values, paths to secret files, or secret material.
- Run the existing digest-pinned `build --pull` and `up -d` path, verify the volume identity is unchanged, then require the normal one-shot/startup, runtime-preflight, and exact application-health gates. On failure, retain deployment state, data, logs, and the already-written evidence for operator-controlled diagnosis/rollback.
- Extend deterministic installer contracts for command shape and every refusal, and add one clean-clone Docker scenario that performs a live install followed by two updates. The scenario uses unique Compose resources through the harness's established Docker-shim isolation, refuses any operator-project collision, and proves the volume identity and source/staged Nautt key digests remain unchanged while startup and health pass after both updates.
- Document the update, backup prerequisite, preserved evidence location/content, secret continuity, verification, and schema-aware rollback boundary in operator-facing docs.

## Affected areas

- `install/update.sh` — new guarded update command.
- `install/test.sh` — executable contract and regression coverage for update behavior.
- `pop/scripts/container-test.mjs` — disposable live install/update/re-run scenario using unique project resources.
- `.gitignore` — exclude the default local rollback-evidence directory from source control.
- `README.md` and `docs/production-runbook.md` — supported invocation and operational/rollback expectations.
- `AGENTS.md` — revise the root `install/` ownership contract because the operator flow gains an update surface.
- `pop/specs/administrative-foundation.md` — at closeout, record the durable guarded-update requirement and delivered slice; no new spec is needed because this extends the existing self-hosted runtime contract.

## Execution front

### Front 1 — guarded operator update

- **Owns:** `install/update.sh`, `install/test.sh`, `pop/scripts/container-test.mjs`, `.gitignore`, `README.md`, `docs/production-runbook.md`, `AGENTS.md`, and the task/card artifacts required by `sync-specs`.
- **May read:** `install/install.sh`, `install/uninstall.sh`, `install/.env.example`, `compose.yaml`, container health/startup helpers, `pop/specs/administrative-foundation.md`, and package scripts used for verification.
- **Must not edit:** `compose.yaml`, `Dockerfile`, `container/`, Prisma schema or migrations, application source, generated files, populated environment/secret files, or any Docker resource belonging to an operator deployment during automated verification. The live harness may create and delete only its uniquely named disposable project resources and temporary clean-clone files.

## Dependencies and sequencing

- No task dependency. Implementation starts only after the critical yolo 003 gate approves this plan.
- Existing installations created by `install/install.sh` are the compatibility baseline. A missing/foreign Compose label, inconsistent database mount, missing/invalid secret artifact, key mismatch, unsupported environment entry, unavailable Docker/Compose, absent rollback input, or incomplete evidence record aborts before build or deployment mutation.
- Run deterministic contracts without a live operator project. Run the integrated scenario only through `pnpm container:test --clean-clone --scenario update`; its shim maps the fixed production identifiers to a unique project, validates the real unique labels before translating them, and cleanup is scoped to that project.

## Durable contracts and spec impact

- Only a named volume whose Compose labels, driver, owning database container, and mount target match the supported installation is eligible; it survives by identity. The update command never issues `down`, `--volumes`, `--purge-data`, volume removal, or clean-clone cleanup.
- Existing source and staged secrets are read-only inputs. The command never generates an encryption key, initial identity, password, or recovery material, and never prints secret values.
- A backup reference acknowledges the runbook's separately created protected logical backup; the command does not claim to create or validate that backup or schema rollback compatibility.
- The pre-update evidence record is immutable rollback metadata, not a backup or automated rollback mechanism. It must exist before image pull/build and remain available whether deployment succeeds or fails.
- `pop/specs/administrative-foundation.md` and the root DOX install contract must reflect the new supported operator surface once delivered. No other product or Nautt behavior contract changes.

## Risks and abort conditions

- **Wrong project, name collision, or fresh host:** abort unless volume labels, local driver, owning `db` container labels, exact mount source, and mount destination all agree with the supported Compose installation; never turn update into installation.
- **Credential loss:** abort on invalid/mismatched Nautt key or incomplete staged/source secrets before Compose mutation.
- **False recovery confidence:** state that backup reference acceptance is not backup integrity validation and that rollback remains schema-compatibility reviewed and forward-migration aware.
- **Unrecoverable release state:** abort before pull/build if the complete secret-free evidence file cannot be written atomically and protected; a backup reference alone is insufficient.
- **Partial deployment:** preserve containers, volume, logs, and pre-update evidence; report diagnostic commands without automatic destructive rollback.
- **Test collision:** the live scenario aborts rather than rewriting, starting, or cleaning the fixed operator project; only unique harness resources may be mutated.
- **Secret disclosure:** arguments, dry-run output, errors, tests, and docs contain no populated password or key.
