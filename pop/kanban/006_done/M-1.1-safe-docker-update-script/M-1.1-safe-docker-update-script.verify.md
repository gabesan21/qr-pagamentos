## Findings

### Blocking

1. **Rollback evidence can falsely label a materially modified checkout as clean.** `install/update.sh:169-170` calls `git status --porcelain --untracked-files=no`. Untracked application, migration, Docker-context, or Compose files can affect the target deployment while the evidence records `target_git_state=clean`. A disposable probe with an untracked build input reproduced the false clean result. This violates the approved target-revision/clean-dirty evidence contract; ignored secrets do not require suppressing all untracked files. `install/test.sh:267-275` checks only that the field exists, not that tracked and untracked changes produce the correct state.

2. **The update preflight accepts malformed callback/API URLs that the installer and root contract reject.** `install/update.sh:62-66` replaces URL parsing with a permissive regular expression. For example, `https://?invalid` matches and proceeds toward Docker mutation even though it is not an absolute URL with a host. The root application contract requires canonical absolute HTTPS without credentials or a fragment, and `install/install.sh` uses `URL` parsing for this boundary. `install/test.sh:284-327` has no malformed update-URL refusal case, so this regression is not protected.

3. **A compatible installation made by a non-UID-1000 Docker user cannot pass Nautt-key continuity.** The installer deliberately stages files as UID 1000, mode `0400`, for the application container. `install/update.sh:110-118` then invokes host-side `cmp` as the current operator. If the same valid installer is run by UID 1001 (or another unprivileged Docker-group user), that operator owns the mode-`0700` directory but cannot read the UID-1000 mode-`0400` staged key, so update fails as if the keys differed. This contradicts the root `install/` contract's Linux/Docker-group boundary and the task's compatible-existing-installation objective. The fixture at `install/test.sh:184-189` changes modes but leaves staged files owned by the current test UID, masking the production ownership edge.

### Suggestions and nits

None. The blocking findings are behavior/contract defects rather than style preferences.

## Environment limitations

- `docker info` failed with permission denied on `/var/run/docker.sock`; no operator Docker resource was changed.
- The unique clean-clone update scenario was attempted and failed without runtime evidence because Docker cleanup also lacked socket access.
- `pnpm check` and the pnpm-wrapped container contract could not acquire the pinned package-manager/store state and attempted blocked registry access. Direct `node pop/scripts/container-contract-check.mjs` passed pin consistency, then failed on registry DNS while inspecting the pinned base image.

## Decision

**Return to 004 (yolo 005 return 1 of 2).** Fix the three blocking update-command contracts and add deterministic regressions for untracked dirty-state evidence, malformed HTTPS inputs, and the non-UID-1000 staged-key case. Re-run the full critical gate afterward; the live clean-clone scenario remains required when Docker access is available.
