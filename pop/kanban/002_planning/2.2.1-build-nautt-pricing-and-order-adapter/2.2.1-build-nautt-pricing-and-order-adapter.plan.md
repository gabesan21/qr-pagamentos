## Route

Add the smallest cohesive server-only boundary under `src/integrations/nautt/`: an exact-decimal validator, a pricing/order HTTP adapter with fixed production origin and injectable fetch/timeout/clock, and owner orchestration that accepts a locally derived owner id, decrypts exactly that owner's key through the existing server-only credential port, and clears the local plaintext reference after each call. The adapter exposes no route and never accepts provider-returned or browser-returned ownership. This permits later authenticated calls to derive the session owner and public checkout to derive the payment-link owner. Do not refactor the delivered webhook adapter unless compilation requires a local shared constant with unchanged behavior.

Use discriminated quote input (`fiat` or `usdt`) instead of two optional amount fields. Validate positive plain-decimal strings and UUIDs before decryption or dispatch; preserve monetary strings without `Number` conversion. A quote response records `expiresAt` as exactly five minutes after the validated response is received using the injected clock. Add a minimal injected `QuoteOwnershipStore` port keyed by `quoteUuid`: `register({ quoteUuid, ownerId, expiresAt })` on issuance and atomic `claimForCreation({ quoteUuid, ownerId, now })` before credential decryption/provider dispatch. The initial in-memory registry accepts a serialized/reconstructed quote reference on the same service/store instance, uses its stored owner/expiry as authoritative, and returns one opaque unavailable error for unknown, cross-owner, expired, consumed, or duplicate-collision records. A fresh registry after restart/another instance therefore fails closed with no provider fallback.

`claimForCreation` checks owner before consuming (so cross-owner attempts cannot burn another owner's quote), then freshness, then marks the quote consumed before decryption. This blocks replay only within the current registry; it is not provider or application idempotency and cannot resolve an ambiguous POST. Decryption/local failure after claim requires a fresh quote. Task 2.2.2 may replace the port with an atomic durable implementation without changing the public `quote(ownerId, input)` / `createOrder(ownerId, quoteReference, input)` contract or the provider adapter.

Expose redacted domain DTOs only: documented quote monetary fields and an onramp order view with provider order UUID, lowercase documented status, exact fiat/crypto/quote values, expiry, payment method, `pix_qrcode ?? qrcode`, and optional `pix_qrcode_url`. Never return raw envelopes, provider user/payer objects, keys, or error bodies. `POST /orders/onramp` is one-shot: every failure after dispatch is opaque/indeterminate and is never retried; `GET` maps provider 403/404 to one not-found/not-owned error.

Discarded: database/admin UUID management (not needed to implement a caller-supplied server contract and not named by the card); configurable base URL/install wiring (could exfiltrate owner keys and contradicts the fixed-origin DOX contract); local order persistence/polling/idempotency (owned by 2.2.2); a broad generic HTTP framework (no demonstrated second compatible protocol).

## Forks

- If g01 finds the delivered webhook code cannot share a constant without behavior churn → leave it untouched and duplicate only the fixed origin/timeout constants in the new adapter.
- If a documented success omits or contradicts a required DTO field → reject it with the operation's redacted adapter error; do not widen the DTO or expose the raw response.
- If order creation fails before `fetch` is called (input/serialization) → return a definitive validation/unavailable error; if `fetch` was called and any response/parse/transport/timeout step fails → return the indeterminate creation error and do not retry.
- If `pix_qrcode` exists → use it as copy/paste; otherwise use documented `qrcode`; if neither exists → keep the field absent rather than inventing a hosted/rendered URL.
- If an optional `deposit_fields`/`additional_infos` value is not a bounded string record/list → reject before decryption/dispatch; do not infer provider-specific fields.
- If `register` sees an existing `quoteUuid` or fails → do not overwrite/reassign ownership; return an opaque quote-unavailable error and never expose an unregistered quote.
- If `claimForCreation` reports unknown (including restart/another in-memory instance), owner mismatch, expiry, or consumed replay → return the same quote-unavailable error before key decryption/dispatch; never ask Nautt to validate ownership as fallback.

## Abort conditions

- Any implementation requires a real Nautt credential/network call or a configurable non-production origin outside injected tests.
- Meeting the adapter contract is found to require schema, admin route/UI, i18n, installer, container, or environment changes; return to 002 to split/re-scope instead.
- A proposed automatic retry of `POST /orders/onramp`, or a claim that an ambiguous provider result is definitely uncommitted.
- Provider documentation needed to parse the minimum quote/order tuple is absent or conflicts beyond the recorded QR/status inconsistencies.
- Execution would expose a key, raw provider body, payer/user data, or hosted-payment-link call.
- Implementing cross-process durability, retry recovery, or application idempotency would require schema/order persistence; stop because 2.2.2 owns that replacement behind the approved store port.

## Red-team

- **Attack that failed:** reuse owner A's quote with owner B, replay a consumed UUID, or present a previously valid quote after restart. The registry keys authoritative owner/expiry/consumption by UUID; mismatch, consumed, and unknown fail opaquely before decrypt/fetch, cross-owner does not burn A's claim, and a serialized clone succeeds only while the authoritative registry entry exists.
- **Attack that passed → patch:** the interrupted draft could redirect decrypted keys through `NAUTT_API_BASE_URL`, expanded into UI/schema/install work, treated ambiguous order creation as ordinarily retryable, and exposed broad provider objects. The route now fixes the production origin, removes all configuration/UI/schema/env work, makes POST outcomes indeterminate after dispatch with zero retry, and returns an explicit minimal DTO.
