## Exact contracts

`normalizeUsername` rejects NUL/CR/LF, strips only leading/trailing ASCII SP/HT, ASCII-lowercases `A-Z`, then requires 3–32 ASCII bytes matching `^[a-z0-9]+([._-][a-z0-9]+)*$`. Thus stored usernames start/end alphanumeric and never contain adjacent separators. Required vectors: ` Admin.User ` → `admin.user`, `abc`, and 32 lowercase alphanumerics pass; 2/33 bytes, non-ASCII, internal whitespace, `@`, separator edges, adjacent/mixed separators, and control characters fail with a generic non-echoing error.

`normalizeOptionalEmail` preserves the previously approved canonical email grammar when a value is present. `null`, `undefined`, empty input, or ASCII-SP/HT-only input becomes `null`; NUL/CR/LF and every nonblank invalid email fail. PostgreSQL stores `email NULL` for absence, permits multiple NULLs, and enforces canonical unique non-NULL values. The rewritten migration names `user_username_key`, `user_username_canonical`, `user_email_key`, and `user_email_canonical`; the username check is `octet_length(username) BETWEEN 3 AND 32 AND username = lower(username) AND username ~ '^[a-z0-9]+([._-][a-z0-9]+)*$'`, while the email check is explicitly `email IS NULL OR (<existing canonical-email predicate>)`. Email is absent from every login lookup/credential input.

Passwords, exact scrypt record grammar/resources/parser, `ADMIN`/`USER`, `ACTIVE`/`DISABLED`, one-to-one credentials, immutable no-FK deployment locator, UUID-only recovery, and least-grant runtime DML remain exactly as previously approved. Safe user DTOs become `{ id, username, email: string | null, role, status, createdAt }` and still omit hashes, session/internal fields, and secrets.

Installer input changes to required `INITIAL_ADMIN_USERNAME` and optional `INITIAL_ADMIN_EMAIL` in ignored `install/.env`. It writes canonical username plus LF to mode-0600 `.install-secrets/initial_admin_username`; it writes either canonical email plus LF or a zero-byte absence marker to mode-0600 `.install-secrets/initial_admin_email`; generated password/recovery files are unchanged. Staging copies both identity files at mode 0400 to `.container-secrets/`, and base Compose mounts `/run/secrets/initial_admin_{username,email,password}`. Neither identity value enters argv, environment, image layers, or logs. Fresh seed creates exactly one active admin with required username and nullable email; reruns resolve only the immutable UUID and do not rename or fill email. Recovery remains password-only, UUID-targeted, and never changes username/email.
