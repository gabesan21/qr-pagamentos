# Plan — [[6.3.3-provision-persistent-media-operations]]

- **Stage:** 002_planning · **Responsible:** planning agent

## Objective and observable result

- Provision one dedicated Compose-owned local volume that is writable only by the non-root application at a fixed `MEDIA_STORAGE_ROOT`, while the rest of the application filesystem stays read-only.
- Make fresh install, compatible update, default uninstall/reinstall, backup, exact-release restore, and recovery preserve the PostgreSQL metadata/media-byte pair or fail closed with recoverable evidence.
- Prove volume identity, filesystem semantics, target-image revision, old-application availability, rollback, default retention, and destructive cleanup in isolated disposable scenarios.

## Fixed operational contract

### Runtime topology and storage root

- The logical Compose volume is `media-data`, uses the local driver, and mounts read-write only into `app` at `/app/media`. `MEDIA_STORAGE_ROOT` is the fixed container path `/app/media`, not an operator-controlled host path. No database or one-shot service receives the media mount.
- The application remains `user: "1000:1000"` with `read_only: true`; its existing cache and `/tmp` tmpfs mounts remain the only other writable filesystems. The production image supplies an empty root plus `staging` and `objects` control directories owned `1000:1000`, mode `0700`, so first-volume copy-up and every restored volume satisfy the 6.3.2 boundary without a root application process, host `chown`, `sudo`, or ownership rewrite.
- Before binding port 3000, startup requires the exact 6.3.2 local-POSIX contract: canonical non-symlink root, private same-device control directories, regular-descriptor/no-follow/exclusive-create behavior, hard-link no-clobber, file and directory sync, bounded descriptor read, and cleanup of the private probe. Failure emits only a redacted stable preflight code, starts no Next.js child, and makes the container unhealthy.
- Compose and the runtime preflight never create arbitrary paths, weaken modes, follow symlinks, accept special files, expose media on `public/`, publish a database port, or place a secret/URL in an image, volume label, manifest, command argument, or log.

### Volume identity and installation

- A supported volume is exactly local-driver, Compose-labeled for the selected project and logical name, mounted at the one documented destination, and private to the expected container identity. Name alone is not ownership evidence. Missing labels, foreign mounts, bind/NFS/plugin drivers, symlinks, wrong UID/GID/mode, or extra controlled entries abort.
- Fresh install creates and preflights the media volume before reporting success. An idempotent installer rerun reuses the same PostgreSQL and media volume identities; it never replaces a non-empty volume or repairs incompatible ownership silently.
- The first update from the integrated 6.3.2 release may adopt an absent media volume only when the existing installer-managed PostgreSQL volume is compatible and a privileged internal query proves there are zero `MediaObject` rows. The target Compose topology creates and preflights the empty media volume while the old app remains healthy. Rows without a supported media volume, or a pre-existing unowned/incompatible volume, are an abort condition.
- Protected deployment evidence records only the target/previous image revisions, Compose project, both volume identities, operation/result, and timestamps. It never records host mountpoints, paths inside the volume, media identifiers/digests, database values, credentials, or secret paths.

### Update, promotion, and rollback

- One per-checkout operation lock serializes update, backup, restore, recovery, and uninstall. Update retains its clean attached protected-upstream, one-reentry, migration-policy, exact-target-SHA, secret-continuity, and no-moving-ref rules.
- Candidate images are labeled with the captured target SHA. Before promotion, the target image runs its media preflight against the exact existing media volume with no public listener and no provider/network/secret access. The old app remains the same healthy container through candidate build, media preflight, bootstrap, migration, and identity seed; its health is rechecked after the additive migration.
- Any failure before app promotion leaves the old app, both volume identities and contents, secrets, logs, and protected evidence unchanged. The updater never retries a failed one-shot automatically.
- Promotion recreates only `app`, using the target-SHA image and the same media volume. Success requires target revision, both unchanged volume identities, runtime database and media preflight records, and exact `/api/health`.
- If the promoted app fails its bounded health window, retain target logs, recreate the captured previous image against the unchanged volumes, and require the old health contract before reporting rollback success. This rollback is application-image-only: it never reverses migration history or restores/rewrites either volume. If old-image recovery also fails, stop, retain both image identities and evidence, and require operator recovery; never erase data to regain health.

### Consistent backup

- `install/backup.sh` accepts an explicit canonical destination outside the checkout, refuses symlinks/overwrite/unsafe permissions, and creates a mode-`0700` work directory with mode-`0600` artifacts. It requires a healthy supported deployment and the shared operation lock.
- A backup is one versioned, checksummed set containing a PostgreSQL custom-format logical dump, a media archive preserving numeric ownership/modes, and a redacted manifest. The manifest binds the exact application revision, Compose project, both volume identities, schema/migration expectation, archive format, sizes, and SHA-256 values; database rows, media names/digests, secrets, and host mountpoints are forbidden.
- To establish one cut, the script stops only `app`, verifies that no application container remains able to mutate the pair, runs the database/media consistency and descriptor-digest inventory check, captures both artifacts, verifies their checksums/readability, atomically publishes the completed set, and restarts the exact previous app container/image. Failure removes only its un-published temporary set, restarts the old app, preserves all deployment data, and reports no success.

### Exact-release restore and recovery

- `install/restore.sh` is explicit and destructive: it requires a reviewed backup path, an exact confirmation token naming the Compose project, a clean checkout at the manifest SHA, locally available images carrying that SHA, and compatible protected secrets. No force/ignore-version/partial/media-only/database-only mode exists.
- Before touching the managed volumes, restore verifies the manifest grammar, every checksum/size, PostgreSQL dump readability, safe archive members, exact release/schema expectation, volume labels, and media ownership/modes. It rehearses the full pair into uniquely named disposable local volumes and runs migration metadata, database role, media inventory/digest, POSIX preflight, and application health checks there.
- After rehearsal, restore stops the deployment, creates a protected automatic pre-restore recovery set, and replaces the contents of the same managed PostgreSQL and media volumes; their Docker identities do not change. It restores database and media as one pair, reapplies exact private ownership/modes, repeats reconciliation and startup gates, and publishes success evidence only after exact health.
- An injected or real failure after managed-volume mutation automatically attempts the pre-restore recovery set. The observable outcome is either the requested pair healthy or the original pair healthy with restore reported failed. If recovery also fails, all backup/recovery artifacts and volumes remain, the application stays stopped, and the operator follows the runbook; no cleanup, second restore, migration reversal, or purge is automatic.

### Uninstall, reinstall, and safe deletion

- Default uninstall removes containers/networks and disposable staged material but preserves both named volumes, protected source secrets including the Nautt key, backup/recovery sets, and the minimum protected deployment identity needed for reinstall. A reinstall must discover and validate the same volumes and key before startup.
- Data purge is a separate explicit confirmation that resolves exact local Compose-owned PostgreSQL and media volume IDs before deletion. It removes both data volumes together and never backups outside the deployment. An ambiguous/missing/foreign target aborts without deleting either volume.
- Only the clean-clone harness may use `down --volumes`. It generates a unique project, records exact labeled disposable resources, and removes only that inventory. Cleanup failure is a failing test, never a reason to broaden a glob or target an operator project.

## Risks and abort conditions

- **Metadata/byte split:** stop mutation, archive and restore the pair together, verify DB-to-media inventory before publication/health, and forbid partial modes.
- **Foreign-volume deletion:** labels, driver, logical name, container mount, project, and exact resolved IDs must all agree before reuse or purge.
- **Restore corruption:** validate and rehearse in disposable volumes before managed mutation; preserve an automatic recovery set and retain everything on double failure.
- **Target/old-image ambiguity:** bind images/evidence to exact SHA; never promote an unlabeled image or discard the previous identity before target health.
- **Filesystem drift:** pre-bind and restore-time POSIX probes fail closed; never repair symlinks, modes, ownership, or special entries in place.
- **Secret/media disclosure:** protected destinations, closed redacted manifests, no host mountpoints/identifiers/digests in logs, and log-redaction assertions.
- **Abort if** consistent logical backup cannot be made without exposing a credential, the target image cannot preflight the same volume while the old app stays available, the restore rehearsal cannot prove exact-release compatibility, or rollback cannot retain an intact recovery set.
