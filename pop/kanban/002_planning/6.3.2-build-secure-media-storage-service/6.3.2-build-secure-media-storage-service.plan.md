# Plan — [[6.3.2-build-secure-media-storage-service]]

- **Stage:** 002_planning · **Responsible:** planning agent

## Objective and expected result

- **Objective:** establish one server-only, owner/purpose-scoped image boundary that validates, canonicalizes, persists, serves, and retires merchant media without exposing filesystem paths or weakening role isolation.
- **Observable result:** future storefront-logo and product-image callers can store and lifecycle opaque media through the service; only activated media is public, protected reads are owner-only, and malformed, cross-owner, missing, non-ready, or unsafe objects share an empty unavailable outcome.

## Strategy and fixed contract

Add one `MediaObject` model and one additive manifest-generated migration after the immutable 19-migration baseline. Metadata records an application-generated UUID, a unique 43-character base64url public identifier, a separate unexposed random storage key, owner, closed purpose (`STOREFRONT_LOGO | PRODUCT_IMAGE`), state (`WRITING | STAGED | ACTIVE | ORPHANED | DELETING`), a non-negative 64-bit `lifecycleRevision`, canonical MIME, stored bytes, decoded width/height, SHA-256 digest, timestamps, and nullable purge deadline. Every create/activate/orphan/delete/reconcile claim carries the observed revision and conditionally increments it with the expected owner, purpose, and state; no state may reset the revision, so stale claims cannot win after an ABA cycle. Database checks constrain closed values, positive limits, revision, and state/deadline consistency; owner/purpose/state and purge indexes support quota and cleanup. Runtime receives only demonstrated table DML.

Use exactly `sharp@0.34.5` as a direct runtime dependency. Before rasterization, a bounded ≤5 MiB buffer is signature-sniffed and passed to Sharp with `failOn: "error"`, `limitInputPixels: 16_777_216`, sequential reads, disabled libvips cache, and one decoder worker; `metadata()` must report JPEG/PNG/WebP, one page/frame, width/height ≤4096, and area ≤16,777,216 before any `rotate()`/WebP `toBuffer()` call. Processing has a five-second Sharp timeout. A committed `media:decoder-preflight` script must assert the exact package version, bundled libvips availability, valid single-frame fixtures, page detection, and header-declared pixel/dimension rejection under the pinned Node image; the Docker builder runs it before `next build`. Failure is a hard build/runtime refusal, never a relaxed limit. Normalize orientation and re-encode once as metadata-free, non-animated WebP; reject canonical output over 5 MiB. SVG, GIF, AVIF, mismatched, truncated, animated, or resource-limit inputs never reach persistence/publication. Recommended consumer targets are 1024×1024 for logos and 1600×1200 for product images; these remain advisory.

The per-owner quota is 256 physical objects and 100 MiB of canonical bytes; `STOREFRONT_LOGO` is additionally capped at 8 objects and 10 MiB. A `WRITING` metadata reservation is committed before any file is created. Every metadata row in every state—including overdue `ORPHANED` and `DELETING`—counts until its staging/final paths are absent, both affected directories have been synced, and only then the row is deleted. Admission first completes a full controlled-root inventory/reconciliation; any untracked file, missing tracked file, unverifiable byte total, incomplete scan, or failed reclaim denies all new creates until repaired. The resulting physical inventory and row totals are rechecked in a retry-safe serializable owner transaction, so delayed cleanup and concurrent writers cannot over-admit. New objects become `STAGED` only after durable publication; same-owner/purpose revision-fenced claims activate or orphan them. Cleanup may claim `DELETING` only after the 24-hour grace and uses the same revision fence; stale `WRITING` and crash residue remain counted until durably reclaimed.

`MEDIA_STORAGE_ROOT` is the sole runtime root. The supported durability target is a local Linux POSIX filesystem (ext4 or XFS semantics) where staging and objects share one `st_dev`, `O_NOFOLLOW`/`O_EXCL`, hard links, file `fsync`, and directory `fsync` work; initialization performs and removes a private probe and otherwise fails closed. The root must be an existing canonical absolute non-symlink directory other than `/`; callers never supply paths. Publication opens a same-filesystem temp with `O_CREAT|O_EXCL|O_WRONLY|O_NOFOLLOW` and `0600`, writes and `fsync`s it, `fstat`s the still-open descriptor, atomically hard-links it to the absent final name (no clobber), and syncs the final directory. It then opens that final name with `O_RDONLY|O_NOFOLLOW`, verifies a regular file, reads from that descriptor to EOF through a hard 5 MiB+1 bound, and requires exact stored length plus SHA-256 equality with the reserved metadata before unlinking the temp, syncing staging, and revision-fencing `WRITING→STAGED`; mismatch remains unavailable/reconcilable. A serving read likewise opens once with `O_RDONLY|O_NOFOLLOW`, verifies a regular descriptor, reads at most 5 MiB+1 to EOF, and checks exact persisted length and SHA-256 before a final state+revision recheck and any response commitment. The verified bounded buffer—not a later pathname stream—is the response body; path/device/size alone never establish identity. Files remain under `0700` controlled directories, outside `public/`. Task 6.3.3 alone owns mounts, installer/update/backup/restore/uninstall, production directory creation/permissions, and environment wiring; its provisioned filesystem must satisfy this declared probe.

Expose a server-only service for create, activate, orphan, owner read, public read, and reconciliation. Add `GET /media/[identifier]`: `ACTIVE` is sessionless; `STAGED` and grace-period `ORPHANED` are readable only by their active merchant owner. This is a revocable identifier contract: every success and failure uses `Cache-Control: no-store`, and an `ACTIVE→ORPHANED` revision change makes the next request unavailable publicly. Success returns only the pre-commit descriptor-read, digest-verified canonical bytes with `image/webp`, `nosniff`, inline disposition, exact length, and no metadata/path. Invalid identifiers, missing or inconsistent rows/files, same-size regular-file or symlink swaps, digest corruption, wrong owner/role, stale revision, and non-readable states return the same empty `404 no-store`; administrators gain no owner capability.

## Risks and abort conditions

- **Risk:** database/filesystem partial failure creates a public partial file or quota drift — mitigate with pre-file metadata reservation, count-until-durable-delete accounting, non-readable states, revision fences, and crash-point tests.
- **Risk:** decoder bombs/polyglots or path links bypass nominal MIME/path checks — mitigate with pre-decode byte bounds, decoder limits, canonical re-encode, no-follow/exclusive access, and regular-file/root containment verification.
- **Risk:** cleanup races with activation/read — every operation carries and increments `lifecycleRevision`; the descriptor open is followed by a state/revision recheck at the read linearization point.
- **Abort if:** the safe migration language cannot express revision/check/grant contracts, the Sharp preflight cannot prove page/pixel rejection before raster work, or the local filesystem probe cannot prove same-device hard-link no-clobber plus file/directory sync and descriptor `O_NOFOLLOW`; record the exact failed fixture/probe as `RECON NEEDED`.
- **Abort if:** completion requires a production volume, mount, backup, installer, or Compose change; mark `blocked: true` for dependency 6.3.3.
