# Plan — [[6.3.2-build-secure-media-storage-service]]

- **Stage:** 002_planning · **Responsible:** planning agent

## Objective and expected result

- **Objective:** establish one server-only, owner/purpose-scoped image boundary that validates, canonicalizes, persists, serves, and retires merchant media without exposing filesystem paths or weakening role isolation.
- **Observable result:** future storefront-logo and product-image callers can store and lifecycle opaque media through the service; only activated media is public, protected reads are owner-only, and malformed, cross-owner, missing, non-ready, or unsafe objects share an empty unavailable outcome.

## Strategy and fixed contract

Add one `MediaObject` model and one additive manifest-generated migration after the immutable 19-migration baseline. Metadata records an application-generated UUID, a unique 43-character base64url public identifier, a separate unexposed random storage key, owner, closed purpose (`STOREFRONT_LOGO | PRODUCT_IMAGE`), state (`WRITING | STAGED | ACTIVE | ORPHANED | DELETING`), canonical MIME, stored bytes, decoded width/height, SHA-256 digest, timestamps, and nullable purge deadline. Database checks constrain closed values, positive limits, and state/deadline consistency; owner/purpose/state and purge indexes support quota and cleanup. Runtime receives only demonstrated table DML.

Accept only non-animated JPEG, PNG, or WebP whose magic bytes and decoder result agree. Reject SVG, GIF, AVIF, multi-frame input, malformed/truncated input, input over 5 MiB, dimensions above 4096×4096, or decoded area above 16,777,216 pixels. Normalize orientation and re-encode once as metadata-free, non-animated WebP; reject canonical output over 5 MiB. Recommended consumer targets are 1024×1024 for logos and 1600×1200 for product images; these are advisory, not aspect-ratio or crop mutations.

The per-owner quota is 256 retained objects and 100 MiB of canonical bytes; `STOREFRONT_LOGO` is additionally capped at 8 retained objects and 10 MiB. Quota includes `WRITING`, `STAGED`, `ACTIVE`, and grace-period `ORPHANED` records and is reserved in a retry-safe serializable owner-scoped transaction so concurrent writers cannot over-admit. New objects are `WRITING` until the durable file is published, then `STAGED`; a same-owner/purpose claim activates one object, replacement/detach makes the old object `ORPHANED`, and cleanup may tombstone then delete objects only after a 24-hour grace. Stale `WRITING`, stale staging files, untracked final files, and retryable `DELETING` objects use the same bounded 24-hour reconciliation rule.

`MEDIA_STORAGE_ROOT` is the sole runtime root. Media calls fail closed unless it is an existing canonical absolute directory other than `/`, with no symlink root or path component. Callers never supply a path or filename: controlled staging/final directories and validated internal storage keys stay beneath the canonical root. Writes use same-filesystem temporary files, exclusive/no-follow handles, mode `0600` under `0700` directories, file/directory sync, and atomic no-overwrite publication; files are never placed under `public/` or served by a static file server. Task 6.3.3 alone owns Compose mounts, installer/update/backup/restore/uninstall, production directory creation/permissions, and runtime environment wiring.

Expose a server-only service for create, activate, orphan, owner read, public read, and bounded reconciliation. Add `GET /media/[identifier]`: `ACTIVE` is sessionless and immutable-cacheable; `STAGED`/grace `ORPHANED` is readable only by its active merchant owner and is `no-store`. Every success streams only the canonical regular file with `image/webp`, `nosniff`, inline disposition, exact length, and no metadata DTO/path. Invalid identifiers, missing rows/files, unsafe file types/links, wrong owner/role, and non-readable states return the same empty `404` with `no-store`; administrators gain no owner capability.

## Risks and abort conditions

- **Risk:** database/filesystem partial failure creates a public partial file or quota drift — mitigate with non-readable transitional states, atomic publication, compensating cleanup, and crash-point tests.
- **Risk:** decoder bombs/polyglots or path links bypass nominal MIME/path checks — mitigate with pre-decode byte bounds, decoder limits, canonical re-encode, no-follow/exclusive access, and regular-file/root containment verification.
- **Risk:** cleanup races with activation/read — require conditional state/version transitions; a stale sweeper cannot delete a reclaimed/activated object.
- **Abort if:** the safe migration language cannot express required checks/grants, the decoder cannot enforce frame/pixel/resource limits before allocation, or atomic no-overwrite publication cannot be proven on the configured root; record the exact failed fixture/filesystem check as `RECON NEEDED` instead of weakening the boundary.
- **Abort if:** completion requires a production volume, mount, backup, installer, or Compose change; mark `blocked: true` for dependency 6.3.3.
