# Plan — [[7.1.1-model-supported-exchange-currencies]]

- **Stage:** 002_planning · **Responsible:** planning agent

## Objective and expected result

- **Objective:** establish one closed, history-safe exchange-currency model for `BRL`, `ARS`, `BOB`, and `COP`, with trusted Nautt UUID mappings and deterministic activation/default-selection behavior.
- **Observable result:** administrators can maintain provider mappings without rewriting an in-use pair; owner product/store callers can select only active supported currencies; existing links and orders keep their original provider meaning; public callers gain no catalog-discovery surface.

## Strategy and fixed contract

Keep the four ISO 4217 alphabetic codes as the only application currency identities; code controls display/selection semantics, while the mapped Nautt `currency_uuid` is the fiat currency identifier and `exchange_currency_uuid` is the provider exchange/payment-rail identifier accepted by `/pricing/panel/buy`. Provider UUIDs remain canonical lowercase UUIDs, server-resolved, and never sourced from browser owner fields, environment variables, or request headers.

Add a versioned supported-currency mapping over the existing immutable-UUID `CatalogCurrencyPair` records rather than repurposing or backfilling historical links/orders. A code may have inactive historical mappings but at most one active mapping; activation/deactivation and replacement serialize in one transaction. Deactivation removes only new-selection eligibility: referenced products, links, order snapshots, and read projections retain their recorded code/pair/UUID meaning. Physical deletion and in-place UUID remapping are forbidden for referenced or historical mappings.

`BRL` is the sole default candidate. A resolver returns the active BRL mapping or an explicit unavailable/no-selection result; it never falls through to ARS/BOB/COP, insertion order, label order, or an inactive row. Tasks 7.1.2 and 7.2.2 own persistence of store/product choices and must require an active mapping only at new assignment time.

## Gaps and preflight

- **RECON NEEDED / BLOCKED:** the repository has no `/exchange-currencies` source or authenticated evidence for the production mappings of all four codes. Before 004, capture a redacted authenticated Nautt `/exchange-currencies` response and record, for each `BRL|ARS|BOB|COP`, the exact fiat `currency_uuid`, exact exchange/payment-rail UUID, operation direction, active support, limits, payment method, and required `deposit_fields`; prove each pair is accepted for panel-buy/onramp semantics. Missing, ambiguous, or multiple eligible mappings require an explicit product/provider decision—never infer from examples or UUID names.
- **Preflight:** wait until active task `6.3.2-build-secure-media-storage-service` has integrated its Prisma migration, then rebase and choose the next sortable migration ID. The Prisma/schema/migration/db-test lane must not overlap its writes.

## Risks and abort conditions

- **Risk:** a wrong UUID direction or undocumented deposit contract creates unusable checkout choices — mitigation: require the exact redacted provider evidence and focused adapter fixtures before implementation.
- **Risk:** deactivation/remapping changes historical meaning — mitigation: immutable pair UUIDs, retained inactive mapping versions, restrictive references, and active-only checks only for new assignments.
- **Abort if:** the safe migration language cannot express the closed-code, single-active mapping, history FK, or least-grant contract; return to 002 rather than raw SQL, backfill, destructive constraint replacement, or historical rewrite.
