# Plan — [[7.3.1-build-merchant-profile-management]]

- **Stage:** 002_planning · **Responsible:** planning agent

## Objective and expected result

- Give an active merchant a separate bilingual `/profile` workspace to change only their normalized username and optional email, or rotate their password after proving the current password.
- Keep the five-entry merchant business navigation fixed. The shell username/account affordance may link to `/profile` outside that primary navigation; `/settings` remains store/provider/locale configuration.
- Preserve username/password-only login, role exclusivity, administrator account management, storefront settings, and every public/sessionless projection.

## Durable behavior

- `GET /profile`, `POST /profile/identity`, and `POST /profile/password` are unprefixed, active-`USER`-only surfaces. The page re-authorizes the cookie principal and sends no active principal to `/login` and active `ADMIN` to `/admin`. Each POST rejects cross-origin first, then re-authorizes before parsing or service work; when no active principal resolves it returns empty `401`, and active `ADMIN` returns empty `403`.
- Profile reads expose only the current merchant's username, nullable email, and a nonnegative profile version. Username and email reuse the canonical identity validators; email stays optional contact data and is never accepted by sign-in, recovery, or lookup.
- Username/email saves use a dedicated profile version, exact expected-version CAS, and one atomic own-user update. A stale/missing target or malformed input is one opaque failure; either username or email uniqueness collision is the same opaque conflict. Successful identity edits increment the version and retain all active sessions; the next request resolves the new principal values.
- Password rotation accepts current password, new password, and exact confirmation. Validation, wrong-current-password, same-current/new password, stale credential state, and unavailable targets share one opaque failure. No submitted password or credential record is echoed, logged, serialized into telemetry, or retained in UI state.
- Sign-in, merchant rotation, and administrator password replacement share one per-user credential/session serialization boundary. Rotation verifies the current credential, conditionally replaces the exact observed credential, and revokes every session in the same transaction. A concurrent sign-in is either created before and revoked, or observes the new credential; it cannot survive by validating the old credential after revocation. Existing unknown-username timing equalization, generic login failure, five-session eviction, and idle/absolute expiry remain intact. Success clears the current cookie and redirects to `/login?password=changed`; no continuity exception exists.
- Profile identity edits never mutate role, status, locale, storefront, checkout policy, Nautt state, product/catalog data, or administrator accounts. Public/storefront/order DTOs remain unchanged and never gain username, email, version, credential, or session data.
- Completion logging adds only literal route templates and the existing safe request-id/status/duration record. It contains no user ID, username, email, submitted value, conflict target, password, credential material, or session token.

## UI and accessibility contract

- Compose two clearly separated cards: identity and password security, each with at most one primary action. Labels stay above controls; username/email use suitable autocomplete, and current/new password fields use `current-password`/`new-password`.
- Applicable states are default, route loading skeleton, opaque error/conflict recovery, changed success, native pending/disabled submission, visible hover/focus, and signed-out-after-password-success. An empty profile state is not applicable because only a resolved active merchant can enter the page.
- Pending state is immediate, announced, and disables all controls in its own form without intercepting or duplicating the native POST. Confirmation is checked server-side. Focus order follows visual order, notices are announced, targets are at least 44 CSS pixels, zoom/reflow works at 320 CSS pixels, and WCAG 2.2 AA/focus/non-color cues use only semantic tokens and owned primitives.
- Browser evidence covers `pt-BR` and `en`, all six themes, and 375/768/1440 widths, with default, validation/conflict, pending, success/logout, keyboard, overflow, target, axe, console, and external-request assertions. Final visual review leaves no severity-2-or-higher issue.

## Risks and abort conditions

- **Credential race:** an old-password sign-in could outlive rotation — mitigate with one shared per-user lock and transactional conditional update/revocation; reject if an executable interleaving test cannot prove both orderings.
- **Identity conflict disclosure:** distinct uniqueness or target errors could enumerate accounts — map database constraints by known names and return only the closed conflict/failed redirects.
- **Role leakage:** a shared profile service or shell affordance could grant `ADMIN` merchant capability — require defense-in-depth `USER` checks and exact route denial tests before any store read or form parse.
- **Migration collision:** this task owns one serialized Prisma lane; abort before 004 if another active worktree edits schema, migration inventory, or database fixtures.
- **Abort if:** the pinned shadcn preflight requires destructive registry replacement, the production evidence environment cannot exercise real authentication/native POSTs, or the session race cannot be made deterministic. Record exact evidence; never simulate PASS.
