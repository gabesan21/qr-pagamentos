# Plan — [[6.2.3-build-role-specific-app-shells]]

- **Stage:** 002_planning · **Responsible:** planning agent

## Objective and observable result

- Replace the two authenticated monoliths with separate bilingual `ADMIN` and merchant `USER` application shells, persistent desktop sidebars, accessible mobile navigation, active-route feedback, role identity, and bounded route workspaces.
- Preserve every current read projection, mutation endpoint, authorization outcome, public/sessionless surface, and unprefixed locale contract. Reorganize already-delivered controls and add honest empty scaffolds without implementing future analytics, directories, catalog/media evolution, or settings persistence.

## Fixed route and capability contract

| Persona | Visible navigation and page route | Delivery in this task |
|---|---|---|
| `ADMIN` | Dashboard `/admin` | localized empty dashboard scaffold; no metrics |
| `ADMIN` | Orders `/admin/orders` | existing re-authorized global read-only projection |
| `ADMIN` | Links `/admin/payment-links` | localized empty scaffold; no projection or owner service |
| `ADMIN` | Users `/admin/accounts` | existing account creation and lifecycle controls |
| `ADMIN` | Settings `/admin/settings` | existing global payment/provider catalog and language controls |
| `USER` | Dashboard `/` | localized empty dashboard scaffold; no metrics |
| `USER` | Orders `/orders` | existing owner-scoped read-only projection |
| `USER` | Links `/links` | existing owner payment-link controls |
| `USER` | Products `/catalog` | existing owner product controls |
| `USER` | Settings `/settings` | existing Nautt, checkout-policy, storefront, and language controls |

- `/products`, `/payment-links`, and `/admin/users` remain POST mutation endpoints; the non-conflicting page URLs avoid changing or shadowing them. Nested order details inherit the matching role shell.
- Every shell layout resolves the active cookie principal and exact role before role-owned data or children. Missing/inactive principals go to `/login`; active wrong-role principals follow the approved `/admin` or `/` redirects. Navigation and scaffolds grant no service capability.
- The administrator shell uses `compact-role-lockup`; the merchant shell uses `merchant-fallback` until a future safe identity slot exists. Add no upload, URL, media read, or merchant-ownership claim.

## Shell and interaction contract

- Separate administrator and merchant RSC adapters own authorization, locale, username/role facts, identity, navigation inventory, and composition. A role-neutral visual frame may receive inert labels, links, identity, and children, but imports no auth/business service and defines no cross-role fallback.
- Limit the client boundary to pathname active-state calculation and mobile disclosure. Match dashboard roots exactly and nested pages on segment boundaries; the active link has `aria-current="page"` plus a non-color signifier.
- Desktop uses a persistent labelled sidebar. At 320/375 the same inventory uses one keyboard-operable 44-by-44 disclosure; hidden copies do not duplicate the accessibility tree. Preserve focus/reading order, visible focus, skip-to-content, reduced motion, wrapping, and no overflow.
- Cover localized default, loading, empty, error/retry, active, hover/focus, disabled/non-applicable, and mobile open/closed states. Scaffolds expose no fake counts, charts, tables, controls, or downstream CTA.
- Reuse the owned inventory and six semantic themes without raw values or page-local variants. The 004 preflight runs pinned local shadcn `info`, `docs sidebar`, and `docs sheet`; additions require `add <component> --dry-run` and per-file `--diff`, never overwrite. Planner CLI registry lookup was unavailable (`EAI_AGAIN`), so this plan assumes no API.

### F03 — Merchant shell and route workspaces

- **Depends on/skills/criteria:** F01; `clean-code-change`, `ui-change`, React

### F04 — Contracts and browser evidence

- **Delivery/owns:** `DESIGN.md`, linked specs, root/UI/brand and justified new

Missing or incompatible input is `BLOCKED`; no front implements a downstream
feature or changes a mutation URL to make a page route convenient.

## DAG and execution topology

1. F01 fixes the inert shell/navigation/i18n contract.
2. F02 and F03 consume F01 in isolated write sets and may run in parallel.
3. F04 integrates source guards, contracts, and fresh evidence after both roles.

Use one execution orchestrator with isolated front branches/worktrees and
central scope integration. This is a multi-skill DAG; do not give the shared
visual/dictionary files to the two role fronts.

## Risks and aborts

- **Cross-role shell or service leakage:** exact-role layout authorization,
  separate navigation inventories, and wrong-role zero-business-call tests.
- **Lost delivered behavior while splitting pages:** characterize existing
  page data calls, forms, actions, notices, and order projections before moves;
  run their route/page tests unchanged or update only structural expectations.
- **Route collision:** retain current POST URLs and use the fixed page map;
  abort if Next reports a page/route conflict rather than moving a mutation.
- **Hydration/bundle drift:** serialize only labels/facts/links to the small
  navigation client; no business DTO, data fetch, storage, or theme state.
- **Abort if:** a scaffold requires an unapproved global/owner projection,
  merchant logo read, new registry source that cannot pass pinned
  info/docs/dry-run/diff, or a durable route/capability change outside the fixed
  matrix; return to 002 with exact evidence.
