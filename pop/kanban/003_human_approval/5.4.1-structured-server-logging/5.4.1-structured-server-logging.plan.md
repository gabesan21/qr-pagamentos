# Plan — [[5.4.1-structured-server-logging]]

- **Stage:** 002_planning · **Owner:** planning agent

## Objective and expected outcome

- **Objective:** Add a dependency-free, server-only JSON request-completion logger and safe request-id correlation to the final API and supported POST-mutation surface.
- **Observable outcome:** Each instrumented request emits at most one newline-delimited JSON record with a bounded opaque request id and closed metadata, and returns that id in `X-Request-Id`; no record contains a request body, raw URL/query, headers, cookie/session value, actor identity, credential, provider data, retry key, capability, verifier, nonce, error text/stack, or `CustomerSnapshotV1` member.

`GET /api/health` is an explicit exception: retain its exact `200`, `Cache-Control: no-store`, and `{"status":"ok"}` response with no wrapper or response-header addition. Logging must not alter Nautt callback bodies/statuses, redirects, cookies, cache directives, origin-guard ordering, authentication, rate-limit outcomes, or service calls.

## Strategy

Create one small `src/observability/server-request-log.ts` seam. It owns an immutable closed record and a wrapper that normalizes `X-Request-Id` with this single contract: accept the entire inbound header value only if it matches `^[A-Za-z0-9][A-Za-z0-9._-]{0,63}$` (ASCII, 1–64 characters); do not trim, case-fold, decode, or otherwise transform an accepted id. This permits examples `a`, `req-42`, `ABC.def_9`, and a 64-character matching value; it rejects a missing/empty value, 65 characters, leading `.`, whitespace, `\r`/`\n` or any other control/non-ASCII character, comma-joined/duplicate values, and every other punctuation. For every rejection, generate `crypto.randomUUID()`; the generated lowercase UUID is the normalized id. Call the existing handler unchanged, add that normalized id verbatim to the returned `X-Request-Id` response header, and write that same normalized id in the completion record. An unexpected throw is rethrown unchanged after one equally redacted failed-completion record. The writer must be best-effort and never become a response failure.

The only emitted shape is `{ timestamp, level, event, requestId, method, route, status, outcome, durationMs }`: ISO timestamp; closed `level`/`event`/`outcome`; a compile-time route template; method; integer HTTP status; and non-negative elapsed milliseconds. `route` is never an actual pathname, so public identifiers and administrator/user ids cannot enter logs. No request value or caught exception is accepted by the logger API. The implementation writes exactly one serialized object line through the platform server logger; it adds no vendor, database, middleware, proxy, persistence, retry, background work, or outbound provider header.

## Risks and abort conditions

- **Sensitive-data leak through convenient context/error logging:** prevent structurally: the logger takes only the closed primitive record and route literals. Never pass `Request`, `FormData`, JSON, params, errors, Prisma/provider objects, or response bodies to it.
- **Client-controlled id becomes log injection, unbounded data, or a secret echo:** accept only the whole-value `^[A-Za-z0-9][A-Za-z0-9._-]{0,63}$` grammar (1–64 ASCII characters, no whitespace/control characters); otherwise generate a UUID. Do not trim or partially salvage an invalid header, and never echo or log it. Echo and log only the single normalized accepted-or-generated id.
- **Instrumentation changes opaque/protocol behavior:** mutate only the additional response header on eligible responses; preserve status/body/cache/cookies/location and rethrow failures. Health is excluded entirely.
- **Dependency collision:** do not implement logger-side rate controls, public-page handling, or origin checks. If the final 5.3.2 plan/integration changes its surface or requires an incompatible response header, return to 002 with the precise conflict.
