## Route

Implement four groups in order. First establish small, test-first identity/password primitives. Email canonicalization rejects NUL/CR/LF, strips only leading/trailing ASCII SP/HT, maps ASCII `A-Z` to `a-z`, then requires 6–254 printable ASCII bytes and exactly one `@`. Local part is 1–64 bytes from `[a-z0-9.!#$%&'*+/=?^_`{|}~-]`, with no leading/trailing/consecutive dot. Domain is 4–253 bytes, has at least two dot-separated labels, each 1–63 bytes from `[a-z0-9-]` with no leading/trailing hyphen; the final label is 2–63 ASCII letters. Acceptance includes ` Admin.Example+ops@Example.COM ` → `admin.example+ops@example.com`, `a@b.co`, 64-byte local and 63-byte label boundaries; tests reject non-ASCII/internal whitespace, missing/multiple `@`, local dot violations, single/empty domain labels, label-edge hyphens, 65-byte local/64-byte label/one-letter final label, and >254 total bytes. Passwords accept 12–128 Unicode code points with no composition rule; plaintext never crosses persistence/logging/DTO boundaries.

The only accepted credential record is ASCII `scrypt$v=1$N=131072,r=8,p=1$<salt>$<key>` (literal fields/order), where unpadded canonical base64url `<salt>` is exactly 22 characters/16 decoded bytes and `<key>` exactly 43 characters/32 decoded bytes. Hash/verify call Node `scrypt` with `N=131072`, `r=8`, `p=1`, `keylen=32`, `maxmem=268435456`; hashing uses `randomBytes(16)` and verification uses `timingSafeEqual` only after exact parsing. Parser input is at most 128 ASCII bytes and full-matches `^scrypt\\$v=1\\$N=131072,r=8,p=1\\$([A-Za-z0-9_-]{22})\\$([A-Za-z0-9_-]{43})$`, decodes to exact lengths, and round-trips canonical base64url. Any mismatch returns the same invalid-credential result without invoking `scrypt`.

Then replace the fixture-only domain schema with additive `User`, one-to-one `PasswordCredential`, and singleton `DeploymentBootstrap` models through a new immutable migration. `DeploymentBootstrap` has only fixed key `id=1`, original `initialAdminUserId UUID` with **no foreign key**, and `createdAt`; a check fixes `id=1`, and a trigger rejects every update/delete after insert. It is an immutable recovery/audit locator, not an immortal-user or final-active-admin constraint: ordinary user deletion, role change, and disablement remain allowed by this task, leaving the UUID tombstone; recovery aborts if that exact user UUID no longer exists and can never retarget/create. Final-active-admin behavior belongs to 1.2.4. Duplicate canonical email, invalid role/status, orphan credentials, and illegal hash metadata fail at the database boundary; runtime receives only demonstrated DML/sequence grants.

Add one idempotent post-migration identity administration wrapper with explicit `seed` and `recover-initial-admin` modes. The operator must set non-secret `INITIAL_ADMIN_EMAIL=<their real address>` in ignored `install/.env`; absence/blank/multiline/NUL or failure of the g01 canonical-email validator aborts—no address is invented. Installer canonicalizes it and atomically writes exactly one UTF-8 canonical-email line plus LF to ignored mode-0600 `.install-secrets/initial_admin_email`. If absent, it atomically generates `.install-secrets/initial_admin_password` as exactly 32 unpadded base64url characters from `randomBytes(24)`, mode 0600; reruns reuse it and output only its path. Staging copies these to ignored mode-0400 `.container-secrets/initial_admin_{email,password}`, mounted as `/run/secrets/initial_admin_{email,password}`. Compose receives neither value in argv/environment/image/logs.

`seed` locks transactionally. With no users/bootstrap row it creates one active `ADMIN`, credential, then `DeploymentBootstrap(id=1, initialAdminUserId=<created UUID>)`. If the singleton exists it exits 0 only when that UUID still resolves to a user with a credential, without comparing/mutating email, role, or status; users without locator, dangling locator, or ambiguity abort. It does not enforce the later final-active-admin rule.

Recovery never reads email. `install/install.sh --recover-initial-admin` first runs the existing `load_install_env`, distinct-password check, `write_secret_sources`, and `stage_secrets`, so `APP_PORT`, `POSTGRES_ADMIN_PASSWORD_FILE`, `MIGRATOR_PASSWORD_FILE`, `RUNTIME_PASSWORD_FILE`, and `STAGED_SECRETS_DIR` are populated exactly as for deployment. It then creates/reuses mode-0600 `.install-secrets/initial_admin_recovery_password` and stages mode-0400 `.container-secrets/initial_admin_recovery_password`.

Extend the existing helper exactly by adding one path-only assignment before the Docker command while retaining all existing assignments: `compose() { APP_PORT=$APP_PORT POSTGRES_ADMIN_PASSWORD_FILE=$POSTGRES_ADMIN_PASSWORD_FILE MIGRATOR_PASSWORD_FILE=$MIGRATOR_PASSWORD_FILE RUNTIME_PASSWORD_FILE=$RUNTIME_PASSWORD_FILE STAGED_SECRETS_DIR=$STAGED_SECRETS_DIR INITIAL_ADMIN_RECOVERY_PASSWORD_FILE=${INITIAL_ADMIN_RECOVERY_PASSWORD_FILE:-} "${DOCKER[@]}" compose -f "$ROOT_DIR/compose.yaml" -p qr-pagamentos "$@"; }`. Recovery then executes exactly `INITIAL_ADMIN_RECOVERY_PASSWORD_FILE=$ROOT_DIR/.container-secrets/initial_admin_recovery_password` followed by `compose -f "$ROOT_DIR/compose.recovery.yaml" run --rm --no-deps identity-recovery`. Thus the helper forwards the five existing base values plus the recovery path to the two-file invocation; values carried are port/file paths only, never secret contents.

New `compose.recovery.yaml` alone defines top-level `initial_admin_recovery_password.file=${INITIAL_ADMIN_RECOVERY_PASSWORD_FILE:?…}` and opt-in `identity-recovery` (profile `recovery`, `db-ops` target, user `1000:1000`, command `container/identity-admin.mjs recover-initial-admin`, `POSTGRES_HOST=db`, database network, existing `runtime_password`, recovery secret targeted `/run/secrets/initial_admin_recovery_password`, restart no). The base `identity-seed` mounts only runtime/email/initial-password secrets, so normal config/up never parses or requires recovery input. Recovery selects only singleton UUID, aborts if missing, rotates/restores that existing user to active `ADMIN`, and never inserts/renames. After success installer promotes/removes candidate as Round 1 specifies; failure retains it for identical retry. Production orders `db → bootstrap → migrate → identity-seed → app`.

Finally run independent static, database, installer, clean-build, and clean-clone identity scenarios, capture evidence in group notes, and close documentation/spec/DOX drift. Do not add login, sessions, public registration, email delivery, MFA, user-management UI, or generic credential-reset APIs.

## Abort conditions

- Stop and mark blocked if any implementation path requires plaintext credentials in argv, environment, Git, image layers, stdout/stderr, or a reusable database seed.
- Stop if the singleton locator cannot be DB-enforced immutable without restricting user deletion/demotion/disablement, or if recovery can retarget/recreate a missing UUID.
- Stop if runtime identity writes require DDL, schema `CREATE`, migration-table access, object ownership, role membership, or broader grants than demonstrated DML/sequence use.
- Stop if the existing immutable foundation migration would need editing, or if live database/clean-clone evidence remains unavailable after the authorized Docker attempt.
- Return to 002 if password hashing, public APIs, session behavior, or deployment topology must materially differ from this plan or the approved spec.

## Red-team

- **Attack that failed:** changing email/rerunning could retarget recovery. The immutable singleton stores the original UUID without FK; it cannot update, and recovery never reads email or substitutes a user.
- **Attack that passed → patch:** a convenient recovery command could expose a password through argv/environment or silently create a replacement admin. The plan now requires file-only input, redacted errors, exact targeting of the seeded identity, an existing-row update only, clean-clone leakage probes, and abort on ambiguity.
- **Attack that passed → patch:** static schema tests alone could miss grants/default-sequence privileges or Compose ordering. The verification table now independently requires disposable live DB probes and two clean-clone container scenarios; unavailable Docker is an abort after a scoped elevation attempt.
- **Round 1 patch:** exact `scrypt` grammar/resources/parser vectors, deterministic required-email/generated-password paths plus immutable recovery locator, and a single aggregate unit-suite run replace all executor judgment and duplicate verification identified by the critic.
- **Round 2 attack passed → patch:** a marker on `User` created an immortal account and the base Compose could demand an absent recovery secret. It is replaced by an immutable no-FK locator that permits normal user mutation/deletion and aborts on missing UUID; recovery moved to an opt-in override. Exact PATH/version proof and ASCII email grammar close the remaining blind spots.
- **Human-intervention patch:** the raw two-file Compose command omitted base interpolation paths. Recovery now reuses the installer's exact source/staging flow and `compose()` helper, which forwards every existing base port/file-path variable plus the recovery file path; contract tests require successful base interpolation and prohibit secret values.
