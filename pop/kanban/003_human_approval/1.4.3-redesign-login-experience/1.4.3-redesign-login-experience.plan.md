### UX wargame

- **Novice path:** arrive → recognize the username and password labels → use browser/autofill → choose the single “Sign in” action → receive a disabled/busy acknowledgement → either reach the existing redirect or get one generic, retryable recovery message.
- **Cognitive walkthrough:** labels above native fields and a single primary action answer Q1–Q3; the busy label/spinner and alert answer Q4. No password rules, email sign-in, or secondary decision is introduced.
- **Norman check:** semantic controls supply affordance; labels/action text are signifiers; pending and alert states are feedback; label-to-field/form-to-submit order is the mapping; `required`, disabled submit, and generic error are constraints; the familiar username/password form preserves the conceptual model.
- **Design preflight:** this is product UI, not a brand surface. `DESIGN.md` and `globals.css` already provide the named tone, semantic tokens, local font, and dark mapping, so no `PRODUCT.md`, palette, new tone, raw value, animation, or page-specific component is authorized.

### RECON NEEDED

- None. “Bilingual” means dictionary parity/unit coverage for both supported locales; an unauthenticated `/login` stays `pt-BR` until the existing successful sign-in flow persists a preference. Do not add a query, cookie, or locale-prefixed route to force English before authentication.

## Route

Keep `src/app/login/submit/route.ts` and the unprefixed route contract unchanged. Replace only the bare page composition with existing shadcn sources: a restrained `PIX ledger` credential card, labels above native autofill-enabled controls, a generic destructive alert on `?error=invalid-credentials`, and a small client submit control that uses `useFormStatus` to show spinner, `aria-busy`, and disabled states. Use only semantic Tailwind/component classes and local IBM Plex tokens; the page gets no bespoke design system, raw CSS value, remote asset, locale selector, or alternate error detail.

Add focused server-rendered component/page coverage before composition, then add a dedicated production Playwright evidence runner for `/login`. It must make eight current captures (light/dark × 320/375/768/1440), check no overflow, local font, labels, autofill, valid focus traversal, 44px inputs/button, no serious/critical axe issue, and generic recovery alert. Its verifier must bind a human visual review to the exact manifest. Dictionary unit coverage supplies the `en` and `pt-BR` language evidence without weakening the persisted-locale rule.

## Forks

- If the owned primitives cannot compose the required page without a visual override or a new primitive → stop implementation and return to 002 with the exact missing primitive; do not create a login-only variant.
- If `pnpm install --frozen-lockfile` succeeds but the login evidence runner cannot launch Chromium, write captures, or run axe → follow the browser abort condition; no source-only fallback is authorized.
- If a proposed “English login” verification needs a URL prefix, query parameter, or non-persisted locale source → retain dictionary-level bilingual coverage and return to 002 for a product-contract decision instead.
- If testing exposes a route behavior change beyond the existing 303/cookie/locale-seeding contract → restore the route boundary and return to 002; this task redesigns presentation, not authentication behavior.

## Abort conditions

- `pnpm install --frozen-lockfile` cannot produce a usable pinned Playwright/Chromium environment, or `pnpm login:evidence`/its verifier cannot complete → set `blocked: true` with the failing command/output; do not substitute static screenshots.
- The required UI can be achieved only by exposing username/password distinctions, infrastructure detail, secrets, or a new locale route/source → stop and return to 002.
- A change needs schema, session, POST-route, token-source, or shared-component-contract changes not covered by this plan → stop and return to 002 with the discovered contract and proposed task split.

## Red-team

- **Attack that failed:** a polished client submit control could accidentally replace the form action or normalize backend failures. The plan confines the client boundary to the submit button and makes existing route tests plus page action assertions a release criterion.
- **Attack that passed → patch:** “both languages in browser evidence” initially implied an unauthenticated English route, which would violate the persisted-preference and unprefixed-route contract. The patch makes bilingual evidence dictionary/component coverage and keeps browser evidence on the actual unauthenticated `pt-BR` route; any different product decision returns to 002.
- **Attack that failed:** reusing `/design-system` captures would not exercise login fields or recovery. The plan requires a separate `/login` manifest, review, and verifier.
