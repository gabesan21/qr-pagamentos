- **Original request:** the plan directly addresses the requested merchant-only profile area for username, optional contact email, and current-password-proven rotation. It correctly preserves username-only login, role-exclusive panels, administrator and storefront boundaries, complete session revocation, and a secondary profile affordance outside the fixed five-entry merchant navigation.
- **Resolved strengths:** origin denial precedes authorization and parsing; page and POST role outcomes are fail-closed; identity writes use a dedicated nonnegative version and exact CAS; password replacement and session deletion are atomic; sign-in, merchant rotation, and administrator replacement converge on one per-user serialization boundary with deterministic interleaving coverage; the additive migration, runtime grants, bilingual owned-component UI, and sequential F01→F02→F03 topology are proportionate. The current active media-operations worktree does not touch Prisma, migrations, or database fixtures, so the serialized migration lane is presently free.
- **Concrete blockers:**
  1. The current-password side-channel contract is missing. The plan preserves unknown-username timing work for login, but it requires only an opaque response for wrong, malformed, missing/stale, or same-password profile cases. In the current implementation, malformed plaintext or a malformed credential record can return before scrypt, so response equality alone does not prove the requested current-password timing property. State which profile-rotation failure classes must execute the same bounded credential-verification work, require current-password proof before any replacement, and add deterministic instrumentation/interleaving evidence (not a flaky wall-clock threshold) that would fail on an early-return path.
  2. The mutation wire contract is not closed enough to verify. Password success is fixed as `/login?password=changed`, but identity success, identity conflict/failure, and password failure have no exact relative redirect vocabulary. AC1 merely says redirects are “closed,” allowing implementation-chosen outcomes and tests to approve themselves. Define the complete finite redirect set and map every validation, CAS, uniqueness, proof, and unavailable outcome to it without echoing submitted or conflicting data.
  3. AC4, AC6, and AC7 are not executable as written. “Deterministic interleaving tests,” “focused page/component/dictionary tests and pinned shadcn preflight record,” and “focused production profile evidence command and its manifest verifier” name neither commands nor artifact/verifier paths. Replace these placeholders with exact commands (and the run-bound manifest/review paths or naming contract), including both sign-in/rotation and admin-reset/rotation lock orderings, all six themes × two locales × three widths, native POST state assertions, and the verifier that rejects stale or incomplete evidence.
