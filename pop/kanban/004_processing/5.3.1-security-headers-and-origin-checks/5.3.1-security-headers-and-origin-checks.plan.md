# Plano — [[5.3.1-security-headers-and-origin-checks]]

- **Etapa:** 002_planning · **Responsável:** agent planejador

## Objetivo e resultado esperado

- **Objetivo:** (a) serve a minimal response security-header set from `next.config.ts` `headers()` (no middleware); (b) reject cross-origin POSTs on cookie-authenticated mutation routes with the existing opaque empty `403`, before any auth/service work.
- **Resultado observável:** every response carries the header set (verifiable via the config unit test and build); a mutation POST with a missing/mismatched `Origin` gets an empty `403` while same-origin posts keep working; the Nautt webhook and the sessionless public checkout are untouched.

## Estratégia

Headers: static `headers()` entry for `/:path*` with `X-Content-Type-Options: nosniff`, `X-Frame-Options: DENY`, `Referrer-Policy: strict-origin-when-cross-origin`, `Strict-Transport-Security: max-age=63072000; includeSubDomains`, `Permissions-Policy: camera=(), microphone=(), geolocation=()`. **CSP is deferred with reason:** Next 16 App Router injects inline hydration scripts, so a strict `script-src` without per-request nonces breaks the app, and nonces require middleware (forbidden by the epoch decision); a CSP with `unsafe-inline` is security theater. Revisit if/when a nonce-capable path exists.

Origin check: one shared helper (e.g. `src/app/origin-guard.ts`) returning an empty `403` (or `null` to proceed), called at the top of every cookie-authenticated POST handler, before body parsing and service calls — same fail-closed family as `ownerProtectedMutationResponse`/`protectedMutationResponse`. Rule: require `Origin` present, parse as URL, compare its host (hostname+port) against `Host`, falling back to `X-Forwarded-Host` when present (documented TLS-proxy topology); missing, malformed, or mismatched → empty `403`. Cookie-authenticated mutations (browser form posts, which always send `Origin` on POST): `login/submit`, `logout`, `language-preference`, `products`, `payment-links`, `payment-links/[id]`, `checkout-policy`, `nautt-credentials`, `nautt-credentials/reset`, `nautt-credentials/register`, and all `admin/*` POSTs (`users`, `users/[id]/{password,status,role,nautt-credentials}`, `payment-settings`, `catalog/{payment-methods,currency-pairs}(,[id])`). Never touch: `api/nautt/webhooks` (provider callbacks have no browser `Origin`), `api/payment-links/[identifier](, /checkout, /checkout/status)` (sessionless capability-based public), `api/health`, and the `[lang]/*` 404 stubs. `SameSite=Lax` session cookies already block cross-site POSTs; the origin check is defense-in-depth, not a contract change.

## Áreas afetadas

- `next.config.ts` — gains the `headers()` export entry.
- `src/app/origin-guard.ts` (new) — shared same-origin guard helper + colocated unit test.
- Cookie-authenticated POST handlers listed above + their `route.test.ts` — wire the guard and cover reject/accept.
- `AGENTS.md` application contract — record the header set and the origin-check invariant on mutation routes (closeout, rule 5).

## Frentes de execução

### F01 — Security headers via next.config

- **Entrega:** `headers()` in `next.config.ts` serving the five-header set on `/:path*`, with a unit test importing the config and asserting the exact set.
- **Escopo:** response headers only; no CSP (deferred, reason above), no middleware.
- **Owns:** `next.config.ts`, `next.config.test.ts` (new, root — vitest default include covers it).
- **May read:** `vitest.config.ts`, `Dockerfile`/`compose.yaml` for TLS context.
- **Must not edit:** any route handler, `container/`, `install/`, middleware (do not create one).
- **Depends on:** nenhuma.
- **Skills:** [[.agents/skills/clean-code-change|clean-code-change]] — *use para a alteração de config e seu teste*.
- **Critérios:** 1, 4.

### F02 — Origin verification on cookie-authenticated mutations

- **Entrega:** shared origin-guard helper wired into every cookie-authenticated POST route; cross-origin/missing-Origin POSTs get empty `403` before any service call; focused route tests.
- **Escopo:** only the cookie-authenticated mutation routes listed in Estratégia; helper + wiring + tests.
- **Owns:** `src/app/origin-guard.ts` (+ test), the listed `src/app/**/route.ts` handlers and their `route.test.ts`; `src/app/owner-guard.ts` / `src/app/admin/guard.ts` only if the wiring reads better through them.
- **May read:** `src/app/relative-redirect.ts`, `src/auth/authorization.ts`, webhook/checkout routes (read-only, to preserve their contract).
- **Must not edit:** `src/app/api/nautt/webhooks/**`, `src/app/api/payment-links/**`, `src/app/api/health/**`, `src/app/[lang]/**`, `src/checkout/**`, `src/integrations/nautt/**`, `next.config.ts` (F01).
- **Depends on:** nenhuma (independent files; same executor runs F01 first to keep the diff reviewable).
- **Skills:** [[.agents/skills/clean-code-change|clean-code-change]] — *use para helper, wiring e testes*.
- **Critérios:** 2, 3, 4.

## Riscos e condições de aborto

- **Risco:** CSP breakage — mitigação: CSP deferred with recorded reason (Next inline hydration scripts + no-middleware decision); no CSP shipped.
- **Risco:** webhook regression — mitigação: webhook route is in `must not edit`; its existing route test stays green in the aggregate gate.
- **Risco:** false rejections behind the TLS reverse proxy if it rewrites `Host` — mitigação: compare against `Host` with `X-Forwarded-Host` fallback; document the proxy requirement in the runbook task 5.5.1 (RECON note above).
- **Risco:** false rejections from non-browser clients — mitigação: all guarded routes are browser form posts (modern browsers always send `Origin` on POST); API/provider surfaces are excluded by design.
- **Abortar se:** wiring the guard requires touching the webhook or public checkout contract — sinalizar `blocked: true` com o conflito exato.
