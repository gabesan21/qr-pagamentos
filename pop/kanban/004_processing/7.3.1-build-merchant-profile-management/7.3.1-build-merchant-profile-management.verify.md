## Blocking findings

1. **Blocking — native mutation contract is replaced by an intercepted fetch.**  
   `src/app/profile/profile-form.tsx:28-46` calls `event.preventDefault()`, constructs `FormData`, and dispatches `fetch(form.action, ...)` for every identity/password submission. The approved plan and `DESIGN.md` require native POSTs without interception or duplication; this changes navigation, redirect, cookie, progressive-enhancement, and failure semantics in the JavaScript path. `tests/profile.evidence.spec.ts:141-190` proves only that one POST happened, while `scripts/verify-profile-evidence.mjs:56-59` labels that request “native” without checking how it was dispatched. Repair the pending state so the browser performs the native submission, and add a source/runtime assertion that fails on `preventDefault`/`fetch` interception.

2. **Blocking, UI severity 3 — the English password-change completion state is Portuguese.**  
   `src/app/login/page.tsx:12` always loads `defaultLocale` (`pt-BR`). Consequently, `artifacts/profile/20260724074444/interaction-en-password-changed-login.png` contains Portuguese copy and is byte-identical to the `pt-BR` capture (both SHA-256 `e1038062395a21c3758a2d4c1c8aa78738401599275a8bb65800502820483f43`). This violates the bilingual interaction/state requirement and AC6/AC7. The evidence test names the capture as English but never asserts localized visible copy, and the verifier checks inventory/hashes rather than locale semantics. Preserve a safe locale through the signed-out completion boundary without expanding the closed redirect vocabulary, and make the evidence fail when localized captures render the wrong dictionary.
