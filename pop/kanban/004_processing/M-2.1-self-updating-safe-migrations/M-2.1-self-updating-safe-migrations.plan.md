# Plan — [[M-2.1-self-updating-safe-migrations]]

- **Stage:** 002_planning · **Owner:** planning agent · **Revision:** return 1

## Objective and expected result

- **Objective:** make `install/update.sh` pull and deploy the latest revision of its tracked remote branch with no backup/release inputs, while making destructive migrations inexpressible and unverifiable migration SQL undeployable.
- **Observable result:** bare `install/update.sh` reaches the exact upstream commit, preserves the installed PostgreSQL volume and Nautt key, validates the pulled migration set offline, runs a fresh migrate container before the commit-bound app image, and leaves the old app running if pull, policy, build, or migration fails.

## Strategy

Require a clean checkout on an attached branch with one upstream, then run explicit non-rebasing `git pull --ff-only`. Capture the resulting immutable commit SHA, require both local `HEAD` and the just-fetched upstream value to equal it, and re-exec `install/update.sh` exactly once with that SHA in an internal handoff. The re-entered process accepts neither a moving ref nor a second handoff and rechecks `HEAD` before policy, build, and deployment. This guarantee assumes the configured upstream is protected and admits commits only through the required migration/quality gates; pulling arbitrary untrusted code cannot be made safe by code from that same pull.

Replace the live-database probe with a closed migration language. The current 19 immutable migration directories and exact `migration.sql` SHA-256 values form a permanently pinned baseline. Every later directory, strictly ordered after the baseline, must contain a versioned canonical JSON manifest and `migration.sql`; the manifest supports only data-preserving operations: create table, add column (nullable or non-null with a typed constant default), create index, add/validate PK/unique/FK/check constraints through typed fields/AST, and grant/revoke privileges. Identifiers, types, literals, checks, referential actions, and privileges use closed schemas with no raw SQL/expression escape. There is no delete, drop, truncate, rename, type-change, update, backfill/data-rewrite, transaction-control, function/trigger/extension, concurrent/nontransactional, or arbitrary-command operation.

A dependency-free `pop/scripts/migration-policy.mjs` emits the only accepted SQL from canonical `migration.safe.json`; `prisma/migration-policy-baseline.json` pins the baseline IDs/hashes. Its verifier fails unless the baseline is exact, every non-baseline directory has one canonical manifest, its ID is newer and ordered, unknown files/fields/operations are absent, and regenerated bytes exactly equal `migration.sql`. It validates the entire repository set after the pinned baseline; the migrate wrapper defines pending as repository IDs lacking a successful `_prisma_migrations` row, rejects applied IDs absent from the repository or checksum mismatches, and lets Prisma deploy only that set. Thus already-applied future migrations remain policy-checked and immutable on every update.

After pull/reexec, invoke `docker run --rm --pull=never --network none --read-only --user <operator> -v <root>:/workspace:ro -w /workspace <pinned-node-helper> node pop/scripts/migration-policy.mjs verify`; pass no environment, secret, or database mount, so host Node/pnpm is unnecessary. Only then build db-ops/app images labelled with the captured commit. Keep the old app running while a newly created migrate container performs normal Prisma deploy. Its wrapper records repository/pending ID digests, then proves each formerly pending row has non-null `finished_at`, null `rolled_back_at`, a matching checksum, and no failure before success. Only then may identity seed run and the target app be recreated; inspected image labels and health evidence bind migrate/app to the captured commit. A migration failure retains the old app, data, volume, key, logs, and evidence.

## Risks and abort conditions

- **Trust boundary:** a compromised/unprotected upstream can replace updater and verifier — abort operational claims unless branch protection and required gates are in force; record this prerequisite without adding operator arguments.
- **Handoff drift:** upstream may move after pull — bind all later work to the captured SHA; remote movement affects only a later invocation.
- **Migration compatibility:** additive DDL can still fail on locks/constraints — use the normal Prisma execution once, retain the old app on failure, and never run candidate SQL as a production probe.
- **Abort if:** dirty/untracked/detached/no-upstream/ahead/diverged/non-FF/remote failure; handoff mismatch/reentry; baseline/manifest/byte-policy failure; missing offline helper; ownership/key/revision-image mismatch; failed/incomplete migration metadata; or target app would be touched before migrate success.
