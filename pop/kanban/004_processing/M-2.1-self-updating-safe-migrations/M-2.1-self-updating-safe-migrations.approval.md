# Approval — [[M-2.1-self-updating-safe-migrations]]

## Round 1 — 2026-07-22

### Decision brief

- **Delivery:** a zero-required-argument updater that fast-forwards to its tracked upstream revision, preserves the installed volume/key, and deploys only migrations that cannot discard existing application data.
- **Strategy:** re-exec the pulled updater, semantically probe pending SQL in a rollback-only live-database transaction, then rerun the ordered Compose startup.
- **Topology:** three sequential fronts: migration policy, updater/deployment, then durable contracts.
- **Primary risk:** arbitrary pending SQL is proposed to run against the live production database before deployment without a complete containment, concurrency, or resource-boundary contract.
- **Plan:** [[M-2.1-self-updating-safe-migrations.plan]] — *review the complete brief and acceptance criteria*.

### Resposta do crítico (yolo)

- **Context:** strong independent.
- **Return:** 1 of 2.
- **Decision:** returned → 002.
- **Blocking reasons and required corrections:**
  1. The live rollback probe does not establish the stated safety invariant. PostgreSQL rollback does not contain every effect available to SQL (for example existing-sequence advancement and session advisory locks), while functions/extensions can introduce further side effects; a final catalog/row snapshot also cannot detect transient `DELETE`/reinsert or trigger effects. The plan must define an enforceable allowed-effect boundary before execution (or move faithful execution to an isolated copy), reject every effect outside it, and add escape/crash fixtures that prove the production database is unchanged.
  2. Probe execution is not faithful or operationally safe yet. Prisma `migrate deploy` applies and records migrations through its own engine/connection behavior, whereas executing SQL inside an outer transaction changes transaction semantics. Snapshotting and executing each migration twice on the live database also doubles DDL/backfill work and locking while the old app can write concurrently. Specify exact pending-set derivation, Prisma-equivalent transaction behavior, application quiescence or lock ordering, lock/statement timeouts, bounded memory/disk/log handling, and kill/interruption recovery; verify concurrent writers and a realistically large fixture.
  3. The pre-build verifier has no runnable production path. The updater promises Docker-only host prerequisites, the existing images contain the previous revision, and F01 produces Node/package code while F02 may not change `Dockerfile`, `compose.yaml`, or container wrappers. Define the exact pulled-revision command/runtime that can reach the existing database as `qr_migrator` before any managed build/recreate, consume the file-backed credential without URL/secret leakage, and work when host Node/pnpm and dependencies are absent. Add this to the owned paths and acceptance evidence.
  4. AC7 proves only that migration metadata exists before the target app becomes healthy. It must prove every invocation creates a new migrate run even on a no-op, every formerly pending row has non-null `finished_at`, that run finishes before the target app container/process starts, and a failed migration cannot start/recreate the target app. Bind the app evidence to the exact fetched revision/image.
  5. Re-executing newly pulled code transfers enforcement to that code. State the protected-upstream/required-gate trust assumption explicitly, bind the handoff to the single fetched commit (not a moving remote-tracking name), and test that the re-entered updater refuses a target mismatch or second re-entry. This need not add release or backup inputs, but the guarantee cannot be presented as independent of upstream trust.

## Round 2 — 2026-07-22

### Decision brief

- **Delivery:** a zero-argument, exact-upstream update whose pulled migration set passes a secret-free offline policy before a fresh migration run and commit-bound app promotion.
- **Strategy:** pin existing history, make future migration SQL exclusively generated from a closed data-preserving manifest language, verify bytes in the existing offline helper, then stage migrate before app recreation.
- **Topology:** three sequential owned fronts: policy/generator, commit-bound deployment, and durable contracts.
- **Primary risk:** the upstream commit is executable authority; the revised plan now states that protected-branch trust boundary and binds every local handoff/image to one fetched SHA.
- **Plan:** [[M-2.1-self-updating-safe-migrations.plan]] — *revision after return 1*.

### Resposta do crítico (yolo)

- **Context:** strong independent.
- **Return:** 1 of 2 historical; no new return.
- **Decision:** approved → 004.
- **Evidence:** all round-one blockers are closed. Candidate SQL is never executed on production during policy checking; the dependency-free generator has a closed schema, pinned baseline, byte equality, and real-PostgreSQL fixtures. The pulled verifier runs in the already-required pinned helper with no network, database, secrets, or host Node/pnpm. The migrate wrapper derives pending/success from Prisma metadata and checks IDs, checksums, completion, and failures around normal `migrate deploy`. AC8/AC9 require a fresh migrate container, migrate-finish before target-app start, exact commit labels, a no-op rerun, and preservation of the prior healthy app on migration failure. AC2 and the trust contract bind re-exec to one fetched SHA and reject mismatch/reentry. The sequential fronts and verification set are proportionate to this critical production boundary.
