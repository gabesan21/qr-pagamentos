## Blocking findings

### B1 — Runtime can physically delete retained categories

`prisma/migrations/20260724010000_owner_product_categories/migration.safe.json:254-267`
and generated SQL line 34 grant `DELETE` on `app.product_category` to
`qr_runtime`. The production store at `src/auth/product-category.ts:136-230`
uses only `SELECT`, `INSERT`, and `UPDATE`; category removal is deliberately
modeled as one-way deactivation. The static and runtime ACL tests then require
the excess grant (`pop/scripts/db-contract-check.mjs:187-194`,
`pop/scripts/db-test.mjs:815`), while the database fixture itself uses it for
test cleanup (`pop/scripts/db-test.mjs:793`).

This violates the approved AC1 least-grant requirement, the Prisma DOX rule that
runtime receives only demonstrated DML, and the durable retention contract. A
compromised runtime could physically delete any unreferenced inactive category.

**Required correction:** remove category `DELETE` from the safe manifest and
regenerate canonical SQL; make ACL checks explicitly deny it. Perform fixture
cleanup through the migrator/admin connection rather than widening production
runtime authority.

### B2 — The concurrency proof does not reach the real transaction boundary

`src/auth/product-category.test.ts:164-177` serializes two calls through a
hand-written in-memory queue; it cannot fail if PostgreSQL isolation, lock
ordering, or Prisma retry/conflict mapping regresses.
`src/auth/product-category-store.test.ts:6-44` supplies already ordered mocked
rows and executes one callback synchronously. The database fixture at
`pop/scripts/db-test.mjs:775-791` performs sequential SQL updates and contains
no competing edit/deactivation or reassignment clients. It also does not run in
the current gate because the pre-existing malformed fixture at line 472 stops
the suite first.

The approved critical plan explicitly requires deterministic concurrent
edit/deactivation evidence and one commit plus one opaque conflict. Inspection
suggests the implementation is sound, but the required regression proof is
absent.

**Required correction:** add a focused test against the actual database
transaction/store boundary with two independent clients for at least
edit-versus-deactivate and competing deactivation/reassignment, proving one
commit, one opaque conflict, no partial product move, and deterministic lock
behavior. Ensure the category database assertions are independently runnable
despite the classified unrelated fixture baseline.
