# Plan — [[5.5.1-production-runbook-and-release-evidence]]

- **Stage:** 002_planning · **Owner:** planning agent

## Objective and expected outcome

Deliver a durable operator runbook and a release-evidence ledger for the completed
MVP on `develop`. The runbook must make the existing single-instance Docker
deployment operable behind a TLS reverse proxy, with explicit configuration,
secret, backup/restore, upgrade, rollback, startup, and health boundaries. The
ledger must bind the release candidate to its integrated dependency evidence and
state exactly what was inspected statically versus what was not run.

This is a documentation/static-evidence task. Per the explicit user command,
no runtime, install, container, browser, database, or aggregate release test is
run or represented as run. The result is a candidate ready for the human's
subsequent operational testing, not a claim of operational validation.

## Strategy and durable artifacts

Create `docs/production-runbook.md` as the authoritative operator procedure and
`docs/release-evidence.md` as the durable, redacted release ledger; add concise
README links to both. They describe the current implementation rather than add
deployment behavior, images, proxy configuration, services, credentials, or
network exposure.

The runbook must establish these non-negotiable operating boundaries:

- The app remains reachable only through Compose's `127.0.0.1:${APP_PORT}:3000`
  binding. A separately operated TLS proxy owns the public listener, certificate
  lifecycle, and forwarding; PostgreSQL and one-shot services never receive a
  public port.
- At the proxy trust boundary, discard client-supplied forwarding headers and
  overwrite `X-Forwarded-For` with the direct client IP. Also overwrite
  `X-Forwarded-Host` with the canonical public host (including a non-default
  port when applicable), set the public `Host`, and forward the TLS scheme.
  This is required both for the process-local rate limiter's trusted-address
  input and for the origin guard's public-host comparison; never append an
  untrusted `X-Forwarded-For` chain or trust a proxy reachable by clients.
- The release configuration separates the three database passwords, maintains
  file-backed secret permissions, keeps `MIGRATION_DATABASE_URL` migration-only
  and `DATABASE_URL` runtime-only, and documents the required canonical HTTPS
  Nautt callback, optional API base, initial-admin recovery, and external backup
  of the Nautt encryption key. Neither document may contain usable values,
  account identity, customer data, or secret-file contents.
- Health is layered: successful `bootstrap`, `migrate`, and `identity-seed`
  one-shots plus `PASS runtime-db-preflight` precede bind; `/api/health` proves
  only the exact application liveness response. The runbook must not turn that
  endpoint into a database/readiness assertion.
- Backup posture includes a protected logical PostgreSQL backup, the associated
  release/configuration metadata, and a separately protected encryption-key
  backup. Restore is an operator-controlled, isolated, destructive procedure:
  stop the affected deployment, preserve evidence, restore only a reviewed
  backup into the intended database volume, then re-establish the startup gates.
  It never uses test-cleanup commands or deletes a production volume by default.
- Upgrade records the source revision, Compose/image identities, configuration
  fingerprint without values, migration expectation, and fresh backup before
  deployment. Rollback restores the previous application/configuration only
  when its schema compatibility is reviewed; migrations are forward-only, so a
  data rollback requires the reviewed backup/restore process. Neither path uses
  `down --volumes` or silently reverses migrations.

The evidence ledger uses one release identifier and has a requirements-to-proof
table: delivered dependency commit/memory, source/doc paths inspected, static
result, runtime command status, and residual risk. It records every skipped
runtime item as **SKIPPED — user directed**, including `pnpm test`, `pnpm build`,
`pnpm check`, `pnpm db:test`, `pnpm container:contract-check`, all
`pnpm container:test` scenarios, `install/test.sh`, installer execution,
Compose startup, proxy reachability, backup/restore rehearsal, and live health
checks. It must never replace these entries with an inferred pass.

## Risks and abort conditions

- A proxy that appends rather than overwrites forwarding headers lets clients
  spoof both limiter identity and the origin-guard public host. The documents
  must call this an abort condition, not a deployment preference.
- A logical database dump without the encryption key cannot restore encrypted
  Nautt credentials; a key backup without access controls is a secret exposure.
  Document both halves and never put either in Git or the ledger.
- A PostgreSQL volume/migration rollback is destructive and cannot be implied by
  application-image rollback. If the prior app is not schema-compatible or a
  restore target is unclear, stop and require an approved recovery decision.
- Skipped runtime proof leaves startup ordering, TLS/proxy behavior, backups,
  restore, and release health unverified. This is an explicit release risk to
  hand to the final human test/merge gate, not a blocker invented contrary to
  the user command.
- If final 5.3.2 or 5.4.1 delivery differs from its approved contract, return to
  002 with the precise source conflict; do not document planned behavior as
  delivered evidence.
