### Yolo critic response

Critical review decision: **returned to 004**.

1. **Blocking — the lease expires before the accepted request budget, so one delivery can have two live provider reads.** `src/integrations/nautt/webhook-intake.ts:14,87-100` fixes the processing lease at 14,000 ms, while criterion 6 explicitly accepts end-to-end work below 14,500 ms. The store intentionally reclaims immediately after expiry (`src/integrations/nautt/webhook-delivery-store.test.ts:18-33` and the disposable-DB equivalent), so a duplicate arriving between 14.0 s and 14.5 s can claim attempt N+1 and issue a second authoritative GET while attempt N is still running. Attempt fencing only rejects N's later finalization; it does not undo the duplicate external read. Make the live lease exceed the maximum permitted processing window with explicit margin (or enforce a hard abort before lease expiry), and add an intake-level held-reconciler test proving a boundary-time duplicate remains `BUSY` with zero second GET while crash recovery still reclaims only after the safe lease.
2. **Blocking — criterion 6's stated timing path is not what its test executes.** `src/integrations/nautt/webhook-intake.test.ts:186-194` performs an immediate mocked reconciliation and in-memory finalization, then arithmetically adds `10_000`; candidate loading also creates already-plaintext buffers. It therefore does not observe the promised 1,000-candidate load/decryption plus injected 10-second GET plus durable finalization, yet the verification table reports that complete path as PASS. Replace this with deterministic timing seams that advance through the provider timeout and include production-equivalent candidate decryption plus Prisma/disposable durable claim/finalize evidence, then assert the same `<14.5s` boundary without relying on a post-hoc constant.
3. **Blocking — criterion 1 does not prove malformed signatures skip constant-time comparison.** `src/integrations/nautt/webhook-signature.test.ts:20-23` tests only `parseWebhookSignature`; the comparison spy is used only for a valid fixed-length signature at lines 25-33. A regression that invokes comparison on malformed length/hex would leave the current suite green despite the criterion's explicit “fails without `timingSafeEqual`” condition. Exercise `verifyWebhookOwner` for every malformed grammar with the comparison spy and assert zero calls (and candidate secret cleanup).
