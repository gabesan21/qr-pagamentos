## Route

Add unlocalized `POST /api/nautt/webhooks`. Cheaply reject `Content-Length > 256 KiB` when present and cancel the untouched body, then consume `request.body` exactly once through a `ReadableStream` reader. For each received `Uint8Array`, add its byte length before copying; if still bounded, retain one byte-for-byte copy so producer buffer reuse cannot alter the signed input. As soon as the total exceeds 256 KiB, cancel the reader, release it, discard retained chunks, and return empty `413` without allocating a combined buffer; only a bounded successful read is concatenated byte-for-byte for HMAC. Never call `request.arrayBuffer()`, `request.json()`, decode, or reserialize before authentication, and do not require undocumented transport headers. Accept only one `X-Nautt-Signature` value matching `sha256=` plus 64 lowercase hex characters; decode to a fixed 32-byte buffer. Only after the bounded read succeeds, load every `ACTIVE` `{ownerId, encryptedWebhookSecret}` candidate, decrypt and HMAC the unchanged bytes, perform a length-safe `timingSafeEqual` for every candidate without early exit, clear plaintext references, and continue only when exactly one owner matches. The route and errors return empty, generic responses and disclose neither owner existence nor candidate count.

After authentication, validate the closed event set, UUID `X-Nautt-Delivery`, and JSON envelope; require header delivery/event to equal body `id`/`event`. Persist only normalized metadata and SHA-256 payload evidence, never raw body, signature, API key, or webhook secret. Add forward-only `WebhookDelivery` and `WebhookDeliveryAttempt` models/migration: unique provider delivery UUID, owner and optional composite owner/order binding, normalized event/order/created-at evidence, closed durable decision/attempt outcomes, processing lease, timestamps, positive attempt number, lowercase payload digest, deterministic constraints/indexes, and runtime DML only.

The delivery store atomically inserts-or-loads the delivery, records every authenticated attempt, and conditionally claims new/retryable/expired-lease work. A processed/ignored duplicate returns the persisted decision with zero provider GET; a live concurrent claim records `BUSY` and returns retryable non-2xx. Verified malformed input records a stable rejected decision when its delivery UUID is usable; missing/invalid headers, oversized bodies, or forged signatures change no state. A valid unknown owner-bound order is durably ignored and acknowledged `204`; a terminal local order is a durable no-op and `204`.

For an actionable active/known-UUID-indeterminate order, extend `ProviderOrderStore` with an owner+provider UUID lookup and call one shared reconciliation operation: decrypt only the matched owner's API key, perform at most one authoritative GET, require the returned UUID to match, and apply the existing owner/provider/state/version CAS. A stale concurrent result or already-final state is success/no-op. Finalize delivery+attempt durably before `204`; provider/DB/finalization failure marks retryable evidence where possible and returns empty `503`, allowing the documented next delivery attempt. There is no internal network retry, no status mapping from notification data, and polling remains complementary.

## Forks

- If Prisma cannot express the optional composite owner/order FK or atomic lease claim (`g01`) → use reviewed migration SQL plus a narrow `$queryRaw`/transaction store behind the same typed port; never weaken ownership or uniqueness.
- If declared or streamed bytes exceed 256 KiB (`g02`) → cancel the body/reader, return empty `413`, and perform zero candidate decrypt/HMAC, parse, persistence, or provider work.
- If zero or multiple secrets match (`g02`) → return the same empty `401`, parse nothing, write nothing, and perform zero provider calls.
- If authenticated JSON/header fields are malformed or contradictory (`g02`) → persist a deterministic rejected outcome only when owner and delivery UUID are known, return empty `400`, and never GET.
- If delivery is processed/ignored/final-order (`g03`) → record duplicate/no-op evidence, return `204`, and perform zero API-key decryptions/provider GETs.
- If an attempt is busy or authoritative GET/CAS/finalization fails (`g03`) → record `BUSY`/`RETRYABLE` evidence, return `503`, and let Nautt schedule the next request; never loop internally.
- If a stale CAS observes a newer/final row (`g03`) → finalize the delivery as processed/no-op and return `204`; never issue a second GET.

## Abort conditions

- Stop and block if verification cannot select exactly one owner before JSON parsing without exposing secrets or using an undocumented header.
- Stop and return to 002 if the 1,000-active-candidate capacity test plus the injected 10-second GET cannot finish durable work inside a 14.5-second test budget; do not silently raise the 15-second contract or impose an unapproved owner cap.
- Stop and return to 002 if concurrent requests can both claim one delivery, a crash cannot be reclaimed after the lease, or runtime ACL needs DDL/ownership/migration-table access.
- Stop and return to 002 if webhook reconciliation would duplicate transition policy, trust notification status, overwrite a final order, or require more than one GET per request.

## Red-team

- **Attack that failed:** forge a body with a real order UUID/status, exploit timing to identify an owner, replay a processed delivery, race two first attempts, or deliver an older response after finalization. Strict pre-parse all-candidate verification, no early exit, unique delivery claim/lease, zero-GET duplicates, and the existing versioned final-state CAS block each path.
- **Attack that passed → patch:** a naive first-match secret scan leaked candidate position and an authenticated retry could cause repeated GETs after success; the route now evaluates every candidate and accepts exactly one, while the durable terminal decision is finalized before `204` and short-circuits every processed replay.
- **Attack that passed → patch:** a dishonest or missing `Content-Length` made `request.arrayBuffer()` consume an unbounded body before the cap could run; replace it with a single-consumption reader that cancels on the first chunk crossing 256 KiB, prove no downstream work, and persist stable rejected evidence only after a bounded body authenticates with a usable delivery UUID.
