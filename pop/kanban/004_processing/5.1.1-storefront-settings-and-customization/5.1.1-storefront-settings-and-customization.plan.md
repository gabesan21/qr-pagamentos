# Plano — [[5.1.1-storefront-settings-and-customization]]

- **Etapa:** 002_planning · **Responsável:** agent planejador

## Objetivo e resultado esperado

- **Objetivo:** Persist per-owner storefront settings — globally unique public slug, bilingual display name (`pt-BR`/`en`), brand accent color, `enabled` flag defaulting to disabled — via one forward Prisma migration, and expose them through a bilingual owner settings card on the existing dashboard (`src/app/page.tsx`) with its own POST route, reusing the checkout-policy guard/redirect pattern.
- **Resultado observável:** An owner opens the dashboard, sets slug/display names/accent color, saves (opaque `?storefront=changed|failed|conflict` redirect), and toggles enablement; defaults keep every storefront disabled until opted in. Durable contract captured in new spec `storefront-and-customization`.

## Estratégia

Follow the task-4.1.1 checkout-policy precedent end to end: owner columns live on the `User` model, the service lives in `src/auth/storefront-settings.ts` with an injectable store (`updateMany` scoped to `id` + `status: "ACTIVE"`), the POST route reuses `requireOwnerFromCookie` / `ownerProtectedMutationResponse` / `relativeRedirect`, and the UI is a client `Card` in the existing owner shell with dictionary-driven copy and the pending-spinner submit pattern.

Slug namespace: slugs only ever appear under the future `/store/[slug]` prefix, so no reserved-word list is needed — validation is format (`^[a-z0-9](-?[a-z0-9])*$`, bounded length) plus global uniqueness enforced by a unique index; a collision maps to the opaque `conflict` notice. Accent color is exactly `#RRGGBB` hex. Display names are nullable (fall back handled by 5.2.1, not here). Database-level checks (slug format, color format) must also exist in reviewed migration SQL per `prisma/AGENTS.md`; constraint/index names deterministic. Enabling requires a valid slug — enforced in the service, documented in the spec. No public `/store/` page in this task; that is 5.2.1.

The new spec is drafted first (F03) so F01/F02 implement against a written contract; `product-scope.md` gains only a Related-specs link (its requirement "optionally publish a product page" already covers storefront).

## Riscos e condições de aborto

- **Risco:** Check SQL de slug/cor divergir da validação da aplicação — mitigação: uma única fonte de formato documentada na spec; `db:contract-check` e `db:test` validam a migration.
- **Risco:** Conflito de unicidade de slug vazar detalhe (distinguir "slug existe" de outros erros) — mitigação: notices opacas `changed|failed|conflict` já são o padrão; `conflict` não identifica o dono.
- **Abortar se:** a migration exigir rewrite de histórico existente ou tocar grants de `qr_runtime` além de DML nas novas colunas — sinalizar `blocked: true` com a saída de `pnpm db:contract-check`.
