## Route

Expose no scheduler, route, or UI. Add one injected server-only operation `recoverWebhookDeliveries(ownerId, localOrderId, selector)`, where the trusted caller supplies owner/local order and `selector` is either `{ kind: "order-history" }` or `{ kind: "delivery", deliveryUuid }`. This mirrors explicit one-order polling: browser fields never choose an owner/provider UUID, and future UI may wrap the service only with session-derived ownership. Return only redacted `unavailable | no-op | reconciled` metadata and newly recorded count.

The supplied repository evidence names both GET endpoints but does **not** establish their HTTP success status, response envelope, list cardinality wrapper, pagination, or failed-record nullability. Therefore production wire decoding is out of this task: define an injected normalized `WebhookDeliveryHistoryPort` with `listOrderDeliveries(ownerCredential, orderUuid, signal)` and `getDelivery(ownerCredential, deliveryUuid, signal)`. The service creates the `AbortSignal`; the selected method is called exactly once with no retry. The exact supported normalized record uses only fields directly evidenced in `raw/webhook.md`'s embedded delivery record: delivery UUID, webhook UUID, order UUID, closed event type, `isDelivered`, `isPermanentlyFailed`, positive attempt number, and valid created-at. Do not include or persist `response_status`/`delivered_at`, because their permanently-failed nullability is not evidenced. Port implementations must fail closed/redacted before returning records; this service never claims a `data` envelope, status code, list wrapper/cardinality, pagination, or other GET wire variant.

Extend persistence with a forward migration. Add a one-row-per-provider-order recovery lease carrying owner, random UUID fence token, and 30-second expiry; claim/reclaim atomically and never hold its transaction across either provider call. Define a project-local accepted-work budget of 25 seconds: ≤10 seconds normalized history port + ≤10 seconds authoritative order GET + ≤5 seconds durable overhead, leaving a 5-second lease margin. Add `INTAKE | RECOVERY` evidence source to delivery and attempt rows; make their payload digest nullable only for `RECOVERY`, and add only directly evidenced provider webhook UUID, delivered/permanently-failed flags, and attempt number. Database checks prove intake still has a lowercase digest while recovery has no digest, is permanently failed, and carries the required provider evidence. Recovered delivery rows bind the exact owner/order and insert a terminal `PROCESSED` or `IGNORED` recovery attempt in one transaction. `ON CONFLICT (delivery_uuid) DO NOTHING` preserves every known ledger row unchanged; no synthetic payload digest or signature is created.

The service first resolves an exact owner+local order with a known provider UUID. For a specific selector, perform a global delivery-UUID lookup classified as same owner/order, conflicting binding, or absent **before** lease claim: either known classification returns one opaque no-op/unavailable result with zero lease claim/release/complete, evidence write, credential decrypt, history call, or order GET. Only an absent specific UUID—or order-history selector—may acquire the fenced lease. Then decrypt only that owner's active credential/registered webhook identity, create a fixed `HISTORY_DEADLINE_MS = 10_000` abort deadline, and call exactly one selected normalized history-port method with its signal. Race the port promise against the application deadline: at 10 seconds abort the signal and return unavailable after matching-token lease release; if the port ignores cancellation, its later resolution/rejection is consumed but cannot resume validation, reconciliation, persistence, or lease completion. The timeout path performs zero order GET/evidence writes and no internal retry.

After an in-deadline return, order-history mode synchronously rejects `records.length > NORMALIZED_HISTORY_MAX_RECORDS`, where the application-local maximum is exactly 128, before record validation/dedupe, order reconciliation, or evidence writes; release the matching lease and return redacted unavailable with no partial persistence. Exactly 128 records may proceed. This is solely a local memory/DB-work bound, **not** a claim about Nautt response cardinality, wrappers, or pagination. Specific mode still requires exactly one normalized record. Validate accepted records against the local provider order UUID and registered webhook UUID; keep only `isPermanentlyFailed: true`, then remove all UUIDs already in the ledger. Nonfailed or all-known evidence is a no-op and performs zero order GETs.

If new permanently failed evidence remains, inspect current local order state. A terminal row records all new evidence as `IGNORED` with zero order GET. An active or known-UUID-indeterminate row invokes existing `reconcileWebhookOrder` once for the order, regardless of delivery count; that operation performs at most one authoritative owner-bound `GET /orders/{uuid}` and the shared CAS. Only after its success/no-op does one fenced transaction insert still-unknown history evidence as `PROCESSED`/`IGNORED` and complete the recovery lease. Failures release the matching lease where possible, persist no partial recovered batch, and return unavailable; a crash is reclaimable after expiry, while stale tokens cannot complete.

## Forks

- If selector is specific and the globally unique UUID is same-bound or conflicting-known (`g03`) → return one opaque result before lease claim/release/complete, evidence write, credential decryption, history call, or order GET.
- If history is empty/nonfailed/all-known (`g03`) → complete the lease as no-op; list mode used one history GET, specific-known used zero, and neither issues an order GET.
- If the local order is terminal after history discovery (`g03`) → atomically record new failed evidence as `IGNORED`; perform zero authoritative order GETs and never reopen state.
- If the local order remains actionable (`g03`) → reconcile once through the existing service, then record every still-unknown failed UUID as `PROCESSED`; never GET once per delivery.
- If another recovery lease is live (`g02`) → return unavailable/busy with zero credential/history/order work; if expired, reclaim with a fresh fence token and preserve prior evidence.
- If order history returns 129 or more normalized records (`g03`) → reject before per-record work, order reconciliation, or evidence writes; release the matching lease, persist no partial batch, and make no provider-cardinality claim.
- If the selected history call has not settled by 10,000ms (`g03`) → abort its application-issued signal, release/fence the matching lease, return unavailable, and ignore any late settlement with zero order GET/evidence writes.
- If the injected port returns a record outside the exact evidenced normalized shape or contradicting requested delivery/order/webhook identity (`g01`) → fail the whole invocation redacted with zero order reconciliation/persistence; never guess a GET envelope, nullability, pagination, or container variant.

## Abort conditions

- Stop and return to 002 if recovery requires a scheduler, browser-supplied owner/provider identity, an undocumented endpoint/query, or provider event/status as payment truth.
- Stop and return to 002 if a known delivery can enter recovery work, concurrent invocations can both call history/order GETs, or a stale lease token can persist/complete evidence.
- Stop and return to 002 if N failed records cause N order GETs, list mode can process more than 128 normalized records, the service cannot enforce cancellation/ignore late history settlement at 10 seconds, a history-port implementation retries internally, or the combined path can hold a database transaction across provider calls.
- Stop and return to 002 if history evidence cannot coexist with intake invariants without a fake digest, weakened owner/order FK, excess runtime ACL, or changes to the shipped 2.3.1 migration.

## Red-team

- **Attack that failed:** supply another owner's local/provider/delivery UUID, replay a ledger-known failed delivery, race two manual invocations, or use old history status to regress a final order. Trusted owner+local lookup, known-before-secret checks, fenced order lease, insert-only UUID collisions, and the existing authoritative CAS block each route.
- **Attack that passed → patch:** naively reconciling each permanently failed record amplified one order into N provider GETs. The service now batches all newly discovered failures under one lease and performs at most one authoritative order GET per explicit invocation.
- **Attack that passed → patch:** forcing history into the intake schema required a fake raw-body digest and could blur HMAC evidence. The forward migration adds typed recovery evidence with a nullable digest only under recovery checks; existing intake rows and constraints remain strict.
- **Attack that passed → patch:** borrowing other Nautt endpoints' `data` envelopes invented GET history cardinality/nullability. Recovery now depends on a normalized injected port limited to the embedded record fields directly evidenced in the supplied webhook document; no production wire decoder is claimed.
- **Attack that passed → patch:** a 30-second lease was asserted without exercising both 10-second network budgets. The deterministic clock matrix holds 10s history + 10s order + 5s durable work, proves duplicates busy through 25s, and permits reclaim only at lease expiry with stale-token fencing.
- **Attack that passed → patch:** removing an unsupported provider cardinality assumption left normalized list work unbounded. List mode now rejects above the application-local 128-record maximum before per-record/DB work, with no claim about Nautt cardinality or pagination.
- **Attack that passed → patch:** a cooperative port could satisfy a documented 10-second budget while an ignoring implementation outlived the lease. The service now issues and races a 10-second abort deadline, releases/fences on timeout, and makes late settlement inert.
