## Reproduction

`pnpm container:test --scenario login` exercised the seeded `admin.user` through the real `/login/submit` endpoint. Invalid credentials returned `303`, the generic invalid-credentials redirect, and no cookie. Valid seeded credentials returned `500` before a session or locale preference was persisted.

Sanitized application evidence identified Prisma `P2010` at the session transaction lock. `pg_advisory_xact_lock` returns PostgreSQL `void`; `$queryRaw` attempted to deserialize that result and rejected the unsupported native type. No generated credential, cookie value, connection string, or usable URL is retained here.

## Characterization

Focused command: `pnpm exec vitest run src/auth/session.test.ts`.

Before the fix, the added regression failed because `acquireUserSessionLock` did not exist and the production adapter still used the result-reading API. After the fix, it verifies the lock statement is sent through `$executeRaw`, which executes the statement without deserializing its `void` result.

## Repair

The session transaction now acquires the same transaction-scoped advisory lock through `$executeRaw`. Lock scope, per-user serialization, session eviction, credential semantics, and the surrounding transaction are unchanged.

Rejected alternatives:

- Catching the route exception and returning invalid credentials would hide infrastructure failure and violate the opaque-authentication contract.
- Casting the lock result would couple the query to an artificial return value when no result is consumed.
- Changing schema, migrations, privileges, or locale persistence was unsupported by the original login exception.
