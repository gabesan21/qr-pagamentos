# 4.1.2-payment-link-order-lifecycle — Plan

## Objective and boundary

Deliver the durable, owner-bound payment-link order record and server-only lifecycle primitives that public checkout will consume. An order is created from an active, unexpired persisted link in one transaction, snapshots the trusted product/currency values and exactly the owner's accepted customer fields, and can make one concurrency-safe successful-payment claim for a single-use link.

This task adds no public route, checkout UI, dictionary copy, browser/customer submission handler, provider quote/onramp request, provider retry, polling/webhook/recovery orchestration, public status capability, or protected order view. Those remain 4.2.1, 4.2.2, and 4.3.1. Therefore `ui-change`/`ui-review` do not apply; the card correctly lists `clean-code-change` + `sync-specs` for 004 and `clean-code-review` + `yolo-critic` for 005.

## Durable model and transition contract

- Add `PaymentLinkOrder`: `id`, `owner_id`, `payment_link_id`, `product_id`, immutable canonical product-price and the two currency UUID snapshots, an applied closed checkout policy, nullable accepted `name`, `email`, `cpf`, and fixed address members (`street`, `number`, `district`, `city`, `state_uf`, `postal_code`, `country`, nullable `complement`), lifecycle state/version, settlement timestamp, and audit timestamps. No address JSON/blob or extra key exists. Database and server validation implement the proposed [[pop/specs/checkout-and-order-lifecycle|Customer snapshot V1]] tuple exactly: `NONE` all null; `NAME_EMAIL` name/email only; `EMAIL` email only; `NAME_EMAIL_CPF` name/email/CPF only; `NAME_EMAIL_CPF_ADDRESS` name/email/CPF plus all required fixed address members. Text is NFC-normalized and Unicode-trimmed before its explicit bounds; CPF persists as validated 11 digits, CEP as eight digits, Brazilian UF is closed, and country is `BR`. No value is coerced through `Number`.
- `PaymentLinkOrder(owner, link, product)` uses composite FKs to the same-owner `PaymentLink(id, owner_id, product_id)` and `Product(id, owner_id)`. The link's owner/product relation remains the source of truth; the direct product relation makes that identity inspectable while snapshots preserve what was payable. Add the necessary composite unique keys; restrict deletion so order evidence cannot be orphaned.
- Add nullable unique `ProviderOrder.payment_link_order_id` with a composite same-owner FK to `PaymentLinkOrder`. Thus an order has zero or one provider order until 4.2.1 attaches it, and a provider order can serve at most one checkout order. Existing provider-only records remain nullable/unattached.
- States are closed: `CREATED` (snapshot persisted; no provider attempt), `PENDING`, `CONFIRMED`, `REJECTED`, `CANCELLED`, `EXPIRED`, `INDETERMINATE`, and `REFUNDED`. 4.2.1 alone may attach a provider order and move `CREATED` to `PENDING`. The proposed fenced `SettlementMapV1` is the only later reconciliation seam: authoritative owner-bound Nautt reads map `new → PENDING`; `processing`/`paid`/`finished → CONFIRMED`; `rejected → REJECTED`; `canceled → CANCELLED`; `expired → EXPIRED`; and `refunded → REFUNDED`. Unknown statuses, webhook notification values, and recovery-history values never map. `PENDING`/`INDETERMINATE` may take every mapped outcome except `REFUNDED`; only `CONFIRMED → REFUNDED` is allowed thereafter; other final states are immutable. `CONFIRMED` is reachable only from the three listed authoritative success statuses and never reopens payment eligibility or erases settlement evidence.
- Persist `PaymentLinkSingleUseSettlement(payment_link_id PK, owner_id, payment_link_order_id UNIQUE, claimed_at)` with composite owner FKs. In one short transaction, the lifecycle store locks/validates the link and order, inserts this row only for `SINGLE_USE`, then conditionally marks the order `CONFIRMED`. A conflict means another order won; reusable links skip this table. No transaction spans provider I/O. This is the concurrency limit—pending, rejected, cancelled, expired, and indeterminate orders never consume a link.
- The future reconciliation adapter passes an internal `SettlementInputV1` only after the existing owner-bound authoritative provider read: local order ID/owner, attached provider-order ID and immutable provider UUID, the observed provider reconciliation version, and one exact mapped authoritative provider status. Its one short transaction requires all of: same-owner attached provider order; matching provider UUID; provider reconciliation version and current mapped status still equal to the observation; local-order version still equal to its observation; and an eligible local state. It then conditionally writes the local state/version and, for `CONFIRMED`, inserts the single-use claim before confirmation. Any stale version, changed status, unattached/cross-owner record, ineligible state, unknown status, duplicate input, or losing claim is a fenced no-op/opaque loser result. It never creates a provider request, retries a provider request, or treats notification/history input as authority.
- Order creation locks and verifies the persisted link/product identity and uses database time: `active = true`, product active, and `expires_at IS NULL OR expires_at > now`; an equal/past expiry fails opaquely and writes nothing. It derives owner, policy, amount, and pair UUIDs solely from those rows; caller input can contain only a validated field snapshot. Expiry remains read-time/create-time enforcement, never automatic link mutation.
- The snapshot is accessible only through a future re-authorized owner/admin projection. It is omitted from all lifecycle, public-link, provider, webhook, recovery, URL, and log DTOs. The later provider adapter must attach the provider order before dispatch; its existing authoritative CAS, webhook intake, polling, and recovery then call the same local settlement transition after reconciliation, never from notification/history payload values. This task changes none of those flows: it persists and tests the seam, not provider calls.

## Risks and abort conditions

- Abort to 002 if the schema cannot prove same-owner link/product/order/provider relationships, one provider order per checkout order, and one single-use settlement claimant without relying on timing or a browser field.
- Do not silently make a single-use link unavailable on order creation: only the successful settlement transaction consumes it. Do not release a recorded claim after a refund.
- Do not attach a provider order, read/decrypt credentials, issue a quote, call Nautt, or change webhook/poll/recovery request behavior in this task. A needed change there is 4.2.1 scope, not a convenience edit.
- Keep PII out of errors, logs, public projections, and tests' diagnostic output. Snapshot validation failures and unavailable/expired links expose no owner/link/order existence facts.
- The address contract is fixed V1: never store arbitrary JSON, unsupported country values, unbounded text, or an unrecognized member. A request carrying address data under any policy except `NAME_EMAIL_CPF_ADDRESS` is rejected without creating an order.
- The settlement seam must not infer status from an event, callback, recovery record, or undocumented provider value. It must fence both provider and local optimistic versions/current states in the confirming transaction; no stale, unattached, or cross-owner input may make an order successful.
- Migration history is immutable; use `MIGRATION_DATABASE_URL` only for Prisma migration work, retain runtime DML-only grants, and update root/prisma DOX only if their durable ownership contracts actually change.
- Docker/container checks, `pnpm db:test`, and browser/runtime evidence are explicitly skipped by user authorization. They are deferred evidence, never reported as passing; static, unit, type, lint, test, and production-build gates remain required.
