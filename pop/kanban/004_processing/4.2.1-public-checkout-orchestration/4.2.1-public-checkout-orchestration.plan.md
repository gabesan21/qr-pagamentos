# 4.2.1-public-checkout-orchestration — Plan

## Objective and boundary

Deliver the sessionless server boundary that turns one active owner-bound payment link and an exact `CustomerSnapshotV1` into one durable Nautt checkout attempt and a minimal public payment capability.  The server—not the request—derives owner, link/product identity, policy, canonical amount, currency UUID pair, credential, quote UUID, provider-order identity, and recovery eligibility.

The public mutation is unlocalized `POST /api/payment-links/[identifier]/checkout`, forced dynamic and `Cache-Control: no-store`.  It accepts only JSON `{ idempotencyKey, customer }`; `idempotencyKey` is an opaque syntactically bounded client retry key scoped to the persisted link, and `customer` must match the link owner's policy-exact `CustomerSnapshotV1` (including no extra members).  It never consumes a session, cookie, account identifier, price, currency/provider UUID, order ID, credential, or browser-selected status.  Invalid input is an empty no-store `400`; malformed/missing/inactive/expired/otherwise unavailable trusted state is one empty no-store `404`; redacted pre-dispatch provider unavailability is empty no-store `503`.

The request's retry key is never stored raw. A committed `(paymentLinkId, retryKeyVerifier)` reservation resolves repeat and concurrent submissions to the same local checkout attempt and authorizes at most one quote/create flow. Its creation transaction locks the active link/product/expiry and records the owner/policy/value snapshot, a keyed verifier of the retry key, and a keyed fingerprint of the exact canonical accepted request (link identity, retry key, contract version, policy snapshot, and normalized customer snapshot). The fingerprint is not serialized request or PII. A duplicate reaches that same attempt only if both verifiers match; a modified policy-valid customer payload, different key, or different link receives the indistinguishable empty no-store `404`, never a capability response or attempt-state oracle. Independently malformed or policy-invalid input remains the normal empty no-store `400`, whether or not its retry key was used before. After its one quote, a second short transaction claims that fresh quote and creates/attaches the same-owner `ProviderOrder` before dispatch. No transaction spans provider I/O, and a `SINGLE_USE` link is still consumed only by the existing successful-settlement transaction. Issue one trusted quote and one `POST /orders/onramp`, never retry after dispatch or release the claim. Complete results move the local order to `PENDING`; every post-dispatch ambiguity is durable `INDETERMINATE`. A known durable provider UUID remains eligible only for the existing one-read authoritative recovery seam; an unknown UUID stays manual, never causes another POST.

The only successful/accepted public body is the exact redacted capability `{ payment: { state, pixCopyPaste?, pixQrCodeUrl? }, statusCapability }`. At reservation time, persist a random server-only capability derivation nonce, capability-key version, expiry, revocation marker, and verifier; never persist the bearer. Re-derive the same high-entropy bearer from that nonce, immutable attempt identity, expiry, and the retained versioned server key, then store/check only its verifier. The recorded key version must remain usable only through its final unexpired capability. This permits durable replay across processes without recoverable bearer storage. `201` represents a complete pending capability and `202` a same-attempt creating/indeterminate capability. A concurrent/duplicate exact request can return the same `202`/`201` DTO only while the persisted capability is unrevoked, unexpired, and the source link remains active/unexpired; its expiry is `min(link expiry when present, reservation time + 24 hours)`. Disabled/expired links and an atomic future owner revocation block reissue. A duplicate never quotes, attaches, recovers, or makes another Nautt POST. The body contains no local/provider/owner/link IDs, customer fields, raw envelope, key, verifier, nonce, or error; none of those values reaches logs, URLs, or cache. A later task may consume the opaque capability for UI/status, but this task adds no public status page/polling route, checkout UI/dictionaries, or owner/admin order view.

## Execution fronts

### F01 — Durable checkout attempt and Nautt orchestration

- **Owns:** the narrowly necessary payment-link-order/provider-order persistence/migration and database-contract assertions, server-only checkout orchestration plus focused unit/store tests, and `src/integrations/nautt/**` only where its typed trusted-call/attachment contract must accept the local checkout attempt.
- **May read:** root/prisma/Nautt DOX, the three linked specs, 4.1.1/4.1.2 memories, existing public-link resolver and provider-order contracts.
- **Must not edit:** public route/UI/dictionaries, protected order views, webhook/poll/recovery execution, generated Prisma output, existing migrations, or provider protocol shapes beyond the required typed attachment seam.
- **Depends on:** 4.1.1 (`0f638c8`) and 4.1.2 (`54e4265`); **input:** identifier plus untrusted retry key/snapshot, otherwise locked persisted link/product/policy/order/provider state; **skills:** clean-code-change, sync-specs; **criteria:** AC1–AC4.

### F02 — Sessionless HTTP boundary

- **Owns:** `src/app/api/payment-links/[identifier]/checkout/**` and focused route/route-boundary tests.
- **May read:** F01's exported request/result contract, root DOX, existing public resolver and unlocalized API route tests.
- **Must not edit:** F01 ownership, Prisma, checkout UI/dictionaries, protected routes/order views, webhook/poll/recovery, or provider adapters.
- **Depends on:** committed F01; **input:** F01's opaque unavailable/invalid/accepted outcomes only; **skills:** clean-code-change, sync-specs; **criteria:** AC5–AC6.

`F01 → F02` is sequential because the route must not invent replay, error, or capability semantics.  No write-safe parallel wave exists.

## Risks, contracts, and acceptance

- **Risks/abort:** abort back to 002 if persistence cannot atomically enforce link-scoped retry-key/request verifiers, a server-only derivation nonce plus capability verifier/version/expiry/revocation, one same-owner pre-dispatch attachment, and durable indeterminacy without provider idempotency. Never expose PII, retry keys, capability/derivation material, or attempt identity in logs/URLs/errors; never turn mismatch, expiry/revocation, link unavailability, credential state, quote/store failures, or provider bodies into distinguishable public facts; never infer recovery from a notification or invent a provider lookup.
- **Specs/DOX:** [[pop/specs/checkout-and-order-lifecycle|Checkout and order lifecycle]] is a draft amendment for this public mutation, retry/capability, and redaction contract; [[pop/specs/nautt-finance-integration|Nautt Finance integration]] and [[pop/specs/catalog-and-payment-links|Catalog and payment links]] retain provider and active-link authority.  Review root, `prisma/AGENTS.md`, and `src/integrations/nautt/AGENTS.md` at closeout; update only contracts whose ownership changed.
