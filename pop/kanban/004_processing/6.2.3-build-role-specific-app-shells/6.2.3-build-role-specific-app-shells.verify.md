## Blocking finding

### B1 — focused skip link is hidden behind fixed shell chrome at every viewport

- **Evidence:** `src/app-shell/app-shell.css:3-5,25-31` gives the skip link,
  desktop sidebar, and mobile header `position: fixed` without a stacking
  contract. The sidebar/header occur later in paint order and cover the
  translated-in focused link.
- **Independent browser probe:** after focusing `.app-shell__skip-link`,
  `document.activeElement` is the link but `document.elementFromPoint()` at its
  center returns `.app-shell__mobile-header` at 375 and 768 pixels and
  `.app-shell__sidebar` at 1440 pixels.
- **WCAG/UX impact:** keyboard users receive no visible indication for the
  first tab stop and cannot discover the promised bypass control. This fails
  WCAG 2.2 AA 2.4.7 Focus Visible and the task's explicit skip/focus contract;
  severity 3 because the control is fully obscured across the supported shell.
- **Evidence gap:** `tests/app-shell.evidence.spec.ts:120-139` calls a DOM-order
  array `focusOrder` but never performs keyboard traversal or verifies that the
  focused element is unobscured. Axe does not detect this stacking failure.
- **Required repair:** establish a token-compliant stacking order that keeps the
  focused skip link visibly above both shell chrome variants, then add a
  production-browser assertion at mobile, boundary, and desktop widths that
  tabs to the link and proves its visible rectangle is not covered. Regenerate
  and re-review evidence if the rendered result changes.
