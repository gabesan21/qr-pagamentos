## Original request first

The delivery establishes the intended non-root app-only media volume, pre-bind
POSIX probe, retained-volume installer lifecycle, paired backup/restore
commands, exact-SHA updater, default retention, paired purge, and operator
documentation. Fresh disposable Docker runs confirm the primary happy paths.

The original request is nevertheless not complete. A normal fresh installer
deployment cannot create the exact-release backup required by the new command,
and the managed restore function can continue after a failed destructive step
and report success from application health alone. Those are direct durability
and recovery failures, not deviations from an optional planning detail.

## Blocking findings

1. **Blocking — fresh installer output is not backup/restore eligible.**
   `install/install.sh:199-202,219-228` neither captures the clean checkout SHA
   nor passes `RELEASE_REVISION`, `APP_IMAGE`, or `DB_OPS_IMAGE`. Consequently
   `compose.yaml:25-30,90-96` builds the normal installer images with the
   `development` revision and `:local` tag. The first operation in
   `install/backup.sh:47-49` then rejects that supported healthy installation
   because its revision is not 40 hexadecimal characters, and
   `install/restore.sh:35-37` could not resolve the required
   `<project>-app:<revision>` image even if a set existed. The `media-backup`
   fixture masks this integration break by directly setting
   `RELEASE_REVISION` before `startHappy`; `install-lifecycle` never invokes
   backup. Bind fresh install to its exact clean release SHA and test
   install → backup → exact-release rehearsal/restore without fixture-only
   image setup.

2. **Blocking — destructive restore failures may be ignored and reported as
   success.** `install/restore.sh:157-177` executes database deletion,
   bootstrap, `pg_restore`, media deletion, and media extraction as unchecked
   commands inside `restore_managed_pair`. Lines 179-180 call that function as
   the condition of `if ! ...` / `if ...`; Bash disables `errexit` for the
   function body in this context. A failed `pg_restore`, failed media cleanup,
   or failed archive extraction can therefore fall through to `compose up`;
   if the still-bootable application passes liveness, line 175 returns zero and
   line 191 publishes `PASS media-restore`. Guard every destructive step
   explicitly, verify the restored database/media pair after managed
   extraction, and ensure any failure reaches the recovery branch.

3. **Blocking — restore does not enforce the promised private media modes or
   protected artifact boundary.** `install/pair-manifest.mjs:7-14,20-41`
   follows artifact symlinks and checks archive traversal/type only; it never
   validates member UID/GID/mode. `container/media-inventory.mjs:23-53`
   authenticates content/count but not file identity or mode, and
   `install/restore.sh:126-133,168-173` preserves archive modes without
   normalizing object files to `0600`. An independent fixture with private
   `0700` root/control directories and one mode-`0777` regular object was
   accepted by `pair-manifest.mjs verify`. Such a pair can pass preflight and
   digest inventory while violating the media filesystem contract. Reject
   symlinked set artifacts and either reject or deterministically normalize
   every restored entry to the exact `1000:1000`, directory `0700`, object
   `0600` contract before rehearsal and managed health.

4. **Blocking — the critical restore/recovery gate does not exercise its
   acceptance contract.** `pop/scripts/container-test.mjs:469-484` covers only
   one checksum-tampered archive and one happy restore. It does not exercise
   unsafe members, wrong SHA/schema/project/volume/confirmation, rehearsal
   teardown refusal before mutation, post-mutation recovery, double failure,
   exact retained recovery artifacts/app-stopped state, or cleanup that refuses
   to broaden beyond inventory. AC6 explicitly requires those observable
   outcomes, and the approved plan says full 005 must inspect every cleanup and
   rollback result. Add deterministic injections/assertions for these paths;
   happy health is not evidence that recovery works.

5. **Blocking — update does not retain evidence for all post-mutation
   failures.** After writing only `phase=database-mutation-started`,
   `install/update.sh:287-298` runs bootstrap and identity seed with `--rm`.
   Failure of either removes its container/log, appends no redacted failure
   result, and does not re-prove old-app health. Only the migrate container is
   retained, and `container-test.mjs:673-685` injects only that migration
   failure. The approved post-mutation contract requires committed
   bootstrap/migration/seed state, failure evidence/logs, and the exact old app
   proven healthy to remain observable. Retain or capture every one-shot
   failure and cover bootstrap, seed, and target-health image rollback.
