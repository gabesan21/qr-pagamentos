## Blocking findings

1. **Blocking — normal install → backup → restore still depends on an
   unrelated global DB-ops image.** `install/install.sh:220-253` now builds
   `${PROJECT}-db-ops:<release SHA>` and verifies its revision, but
   `install/restore.sh:40-56` resolves and verifies only
   `${PROJECT}-app:<release SHA>`. Its managed bootstrap at
   `install/restore.sh:198` and final `compose up` at line 231 therefore use
   the fallback `qr-pagamentos-db-ops:local` from `compose.yaml:24-30,50-52,
   67-69`, without checking its revision. A fresh installation is not required
   to have that global tag; a stale tag may run another release's bootstrap,
   migration, and seed. The fixture sets an exact DB-ops value only in its
   harness-local Compose environment (`pop/scripts/container-test.mjs:445-447`)
   and passes a different `processEnv` to `restore.sh`, so cached global images
   can mask the integration break. Resolve, verify, and export the
   manifest-SHA DB-ops image throughout restore, and prove restore with the
   fallback tag absent or deliberately incompatible.

2. **Blocking — restore's private files work only when the host operator is
   numerically UID 1000.** `install/restore.sh:114-126` creates the candidate
   runtime/Nautt files as the invoking host UID and changes them to mode
   `0400`; lines 159-166 mount them into an application container running as
   `1000:1000`. Lines 219-227 similarly create a host-owned mode-`0400`
   inventory and mount it into a UID-1000 media helper. On any supported Linux
   host whose Docker-group operator is not UID 1000, those helpers cannot read
   the files. The current test host is UID 1000, so the scenario cannot detect
   this. Stage and verify candidate/helper artifacts as exact `1000:1000`
   mode-`0400` files without host ownership repair, and add an ownership
   assertion independent of the invoking UID.

3. **Blocking — the mandatory recovery scenario is red.**
   `pnpm container:test --clean-clone --scenario media-restore` reached the
   paired backup and all pre-mutation adversaries, then failed at
   `pop/scripts/container-test.mjs:605`: after the injected primary and
   recovery failures, the harness attempted to restart the retained pair and
   the exact-release `migrate` service exited 1. It never reached the final
   managed restore or the five restore PASS markers. This contradicts AC6's
   observable requirement that double failure retain an operator-recoverable
   pair with a rehearsed subsequent healthy path. The dated PASS claims in
   `pop/specs/media-storage.md:62`,
   `pop/specs/administrative-foundation.md:127`, and
   `docs/release-evidence.md:34-36,55-59` are not reproducible and must not
   remain PASS until a fresh full scenario succeeds.

4. **Blocking — update seals only a subset of its failure boundary.**
   `install/update.sh:324-343` correctly seals named bootstrap, migration, and
   seed failures, and lines 345-367 handle an app that is created but becomes
   unhealthy. However candidate build/preflight failures at lines 315-321,
   target-checkout failures at lines 332 and 344, failure of the destructive
   `compose up --force-recreate app` at line 345, and final revision/volume
   proof failures at lines 369-371 still exit through `errexit` without a
   sealed redacted result and, after promotion begins, without the required
   previous-image recovery. AC4 and the approved plan cover every failure
   before and after the database-mutation boundary, not only injected helper
   exits. Put these exits inside the same evidence/old-health or image-rollback
   envelope and exercise at least candidate-preflight and app-recreate
   failures.
