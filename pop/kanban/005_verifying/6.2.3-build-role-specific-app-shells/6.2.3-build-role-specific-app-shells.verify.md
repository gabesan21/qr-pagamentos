## Blocking finding

### B2 — relocated Nautt controls lose every result and retry outcome

- **Evidence:** `src/app/nautt-credential-surface.tsx:15-27` is the sole renderer
  for `configured`, `invalid`, `changed`, `recovery`, `reset`, and
  `unavailable`. The surface now lives at
  `src/app/(merchant)/settings/page.tsx:27-40`.
- **Broken result target:** `src/app/nautt-credentials/route.ts:28-35`,
  `register/route.ts:21-27`, and `reset/route.ts:16-21` still redirect every
  result to `/?nautt=...`. `src/app/(merchant)/page.tsx:10-38` declares
  `nautt` but never reads it and renders no `NauttCredentialSurface`.
- **Broken retry target:** `src/app/nautt-credential-surface.tsx:77` still links
  unavailable-balance retry to `/`, whose dashboard performs no onboarding
  status read. The action therefore cannot retry the relocated status surface.
- **User impact:** a merchant is taken away from Settings after every
  credential/register/reset submission and receives no success, validation,
  provider, concurrency, or recovery feedback. This fails Nielsen visibility
  of system status, error recovery, and AC2’s explicit preservation of forms
  and notices; severity 3 and blocking.
- **Test gap:** `src/app/(merchant)/page.test.tsx:12-26` checks only the empty
  dashboard. No integration test binds each Nautt result/retry target to the
  page that renders the corresponding localized state.
- **Required repair:** keep the POST endpoint URLs and protected outcomes
  unchanged, but return each Nautt outcome to the owning Settings page and make
  manual balance retry reload that page. Add tests proving all result codes and
  the retry target reach a re-authorized `NauttCredentialSurface` in both
  locales without disclosing provider or credential data.
