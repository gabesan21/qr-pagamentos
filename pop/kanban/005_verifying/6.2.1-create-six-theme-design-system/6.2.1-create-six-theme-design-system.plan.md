# Plan — [[6.2.1-create-six-theme-design-system]]

- **Stage:** 002_planning · **Responsible:** planning agent

## Objective and expected result

- **Objective:** turn the existing role-neutral `PIX ledger` foundation into one lintable six-theme visual system for QR Pagamentos, with stable identities, semantic token resolution, documented typography and motion, contrast evidence, and deterministic fallback behavior.
- **Observable result:** the same owned shadcn components and `/design-system` specimen render six named personalities—three light and three dark—without raw component styling, business-role coupling, remote assets, or changes to application behavior.

## Strategy

Keep one decisive direction, **PIX settlement desk**: a Brazilian payment-operations workspace derived from the PIX receipt, QR alignment grid, cashier terminal, and clearing ledger. Preserve IBM Plex Sans, factual tabular numerals, ruled hierarchy, restrained corners, and one primary action per section. The signature is the receipt rail aligned to a subtle QR-module rhythm; theme color changes its personality, never the component geometry or information hierarchy. Gradients, ornamental QR codes, generic dashboard cards, serif-led cream editorial styling, purple/neon accents, and color-only status remain banned.

Use DTCG-shaped reference → semantic → component-consumption layers. Shared spacing, type, radius, shadow, focus, and motion tokens resolve before a `theme` modifier; each theme supplies the same semantic color paths. Generate or verify the Tailwind/shadcn CSS projection deterministically so `globals.css` is no longer an independent hand-maintained palette. Components continue consuming semantic names only.

The stable identifiers and personalities are:

| Identifier | Mode | Personality |
|---|---|---|
| `pix-paper` | light, default light | crisp receipt paper, graphite facts, trustworthy PIX teal |
| `cashier-daylight` | light | cool terminal white, deep blue controls, measured cyan-green accent |
| `settlement-sand` | light | warm reconciliation paper, dark umber facts, restrained copper/green action |
| `midnight-clearing` | dark, default dark | graphite clearing desk, pale facts, luminous but non-neon mint |
| `vault-blue` | dark | deep navy custody surface, ice-blue facts, cyan action |
| `terminal-amber` | dark | near-black operator terminal, warm amber facts, muted green confirmation |

Explicit valid theme identifiers override system preference. With no explicit theme, light preference resolves to `pix-paper` and dark preference to `midnight-clearing`; the legacy `.light`/`.dark` aliases map to those defaults. Unknown identifiers fail safely to `pix-paper`. No selection persistence, account preference, new route, locale, or role-specific theme is introduced.

## Risks and abort conditions

- **Risk:** six palettes become six component systems — mitigate by identical semantic paths, shared geometry/type roles, and zero theme-specific component classes.
- **Risk:** authored OKLCH values clip or pass only hard-coded contrast fixtures — require sRGB fallback/gamut validation and compute every asserted pair from resolved tokens.
- **Risk:** the new fallback breaks existing system light/dark evidence — retain explicit default aliases and assert absent, valid, and unknown selection paths.
- **Risk:** visual proof becomes stale or unreviewable — require one run identifier binding all 24 captures, assertions, hashes, and visual review.
- **Abort if:** a complete theme cannot meet WCAG 2.2 AA without changing semantic status meaning, or deterministic projection requires replacing the pinned shadcn/Tailwind stack; record `RECON NEEDED` with the token pair/tool conflict instead of weakening the contract.
