## Blocking finding

### B1 — The resolver describes contexts but does not control generated output

`src/design-system/tokens/resolver.json:3-27` now uses version `2025.10` and
declares real theme and motion modifiers. However, the generator only verifies
that each referenced node exists (`scripts/generate-design-tokens.mjs:11-19`).
It then bypasses the resolver and reads `source.themes`, hard-coded
`pix-paper`/`midnight-clearing`, and `source.motion.full|reduced` directly
(`scripts/generate-design-tokens.mjs:44-55`).

Consequently, changing a valid resolver context reference—for example mapping
`motion.reduced` to the valid `motion/full` source, or mapping the
`pix-paper` context to another valid theme—does not change generated CSS and
does not fail the tests. `scripts/design-tokens.test.ts:37-45` proves only that
context targets exist, then asserts the same hard-coded source paths; it never
resolves a `{theme, motion}` input through `resolutionOrder`.

The resolver also stores projection metadata in a non-DTCG top-level
`projection` member (`resolver.json:28-39`) rather than a namespaced
`$extensions` member. The local DTCG 2025.10 contract permits the standard
resolver members plus `$extensions`; this document is therefore only
DTCG-shaped, not a valid resolver contract.

**Required correction:** implement one resolver function that validates and
applies `resolutionOrder`, selects the requested/default theme and motion
contexts, merges their sources, and resolves aliases/cycles; use that resolved
result for generation and tests. Move CSS projection metadata under a
namespaced `$extensions` member (or a separate non-resolver file). Add a
production-graph test showing that changing a valid context mapping changes the
resolved/generated value.
