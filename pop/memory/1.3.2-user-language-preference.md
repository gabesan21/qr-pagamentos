# 1.3.2-user-language-preference — User language preference

Persisted each authenticated user's nullable `pt-BR` or `en` preference through
an immutable Prisma migration. The first successful sign-in seeds it from a
safe server-side `Accept-Language` negotiation, while an explicit signed-in
selection wins thereafter. Application UI, auth, admin-access, and design-system
routes are now unprefixed; former locale-prefixed surfaces return 404 and
`/api/health` remains unlocalized. Added database, negotiation, concurrency,
authorization, route, dictionary, and native-control coverage. Aggregate,
database, and focused verification passed; Playwright was unavailable, so the
updated 375/768/1440 capture script remains ready for an environment that has it.
